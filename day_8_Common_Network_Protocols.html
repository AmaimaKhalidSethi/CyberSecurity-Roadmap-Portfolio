Day 08: Foundational Mastery

Networking Fundamentals II: Core Network Protocols & Threat Vectors
Incident Responder Perspective 🛡️
Attackers do not break rules; they abuse features. Firewalls, SIEM alert filters, and packet captures rely heavily on analyzing application layer protocols and protocol states. If you can distinguish standard behavior from anomalous protocol abuse, you can stop a security incident before it scales.

1. The Transport Layer Battle: TCP vs. UDP

Before examining application-layer traffic, we must verify how packets are moved. Layer 4 handles this using two completely contrasting delivery mechanisms.

Transmission Control Protocol (TCP)

Mechanics: Connection-oriented. Establishes a guaranteed connection using a strict 3-Way Handshake (SYN ➡️ SYN-ACK ➡️ ACK). Features error checking, packet sequencing, and flow control.

🚨 Attacker Exploitation:

  • SYN Flood Attacks: Flooding a target with spoofed SYN requests, forcing the target to leave thousands of half-open connections waiting for an ACK response, completely exhausting memory pools.

    • User Datagram Protocol (UDP)

    Mechanics: Connectionless. "Fire-and-forget" transmission with zero handshake or overhead. Fast but unverified delivery.

    🚨 Attacker Exploitation:

  • UDP Amplification DDoS: Attackers spoof a victim's IP address when sending requests to open servers (like DNS or NTP). The servers stream massive, amplified response payloads directly to the victim's device, knocking it offline.

    3. Hands-On Protocol Reconnaissance

    Security professionals validate theory by interrogating real-world endpoints using native command-line utilities.

    Deep-Dive Name Resolution with dig

    While nslookup offers basic lookups, dig (Domain Information Groper) extracts detailed DNS zone data. Attackers often abuse DNS text (TXT) fields to store command-and-control payloads or manipulate mail security headers.

    # Query specific TXT records to audit SPF/DKIM authentication rules
    omnisec@soc:~$ dig google.com TXT
    ;; ANSWER SECTION:
    google.com. 3599 IN TXT "v=spf1 include:_spf.google.com ~all"

    Auditing Web Responses & Headers via curl

    Using curl with the -I flag tells the utility to fetch the HTTP headers *only*. This lets you identify server software configurations and see if essential security headers are absent.

    # Interrogate an application layer server to audit security settings
    omnisec@soc:~$ curl -I https://www.google.com
    HTTP/2 200
    content-type: text/html; charset=UTF-8
    x-frame-options: SAMEORIGIN <-- Blocks Clickjacking attempts
    strict-transport-security: max-age=31536000 <-- Forces encrypted connections
