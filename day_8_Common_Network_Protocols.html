Core Network Protocols & Threat Vectors

Attackers don't break rules — they abuse features. Every protocol below does exactly what it was designed to do, right up until someone weaponizes that design. Today is the protocol-by-protocol baseline you'll need before any of the attack or defense tooling in the rest of the roadmap makes sense.

00

Overview

Before examining application-layer traffic, you have to understand how packets move at all.

▸ incident responder framing

Firewalls, SIEM alert filters, and packet captures all depend on telling normal protocol behavior apart from anomalous abuse of that same protocol. If you can make that distinction reliably, you can stop a security incident before it scales — which is the whole reason today starts at Layer 4 and works upward.

01

The transport layer battle: TCP vs UDP

Two completely opposite philosophies for moving the same bytes — and each one's weakness is the direct shadow of its strength.

TCP — Transmission Control Protocol

Mechanics: Connection-oriented. Establishes a guaranteed connection via the three-way handshake (SYN → SYN-ACK → ACK). Provides error checking, packet sequencing, and flow control.

Exploited via: SYN flood attacks.

UDP — User Datagram Protocol

Mechanics: Connectionless. "Fire-and-forget" — no handshake, no delivery guarantee, minimal overhead. Fast, but nothing confirms the other side actually got it.

Exploited via: UDP amplification DDoS.

▸ the shared root of both attack classes

TCP's exploit (SYN flood) and UDP's exploit (amplification) both abuse the exact same property, inverted: TCP gets attacked because it tries to remember state for every connection attempt; UDP gets attacked because it refuses to verify who's actually asking. Neither protocol is "doing something wrong" — they're each doing precisely what they were designed to do.

02

SYN flood, packet by packet

The source names this attack — here's exactly what's happening on the wire and why it works.

Recall the normal three-way handshake:

Client
SYN
Server
Client
SYN-ACK
Server
Client
ACK
Server

Now the attack — the third step never arrives, on purpose, thousands of times per second:

Attacker
(spoofed IPs)
SYN
Server
(nobody)
SYN-ACK
Server
✕
no ACK ever sent
✕

Every server that supports TCP has to allocate a small block of memory — a connection state entry — the instant it sends back a SYN-ACK, because it has to remember that handshake is in progress. With the source IP spoofed, the SYN-ACK goes nowhere useful, and the server sits there holding that half-open connection state until a timeout eventually expires. Flood it with enough spoofed SYNs fast enough, and the server's entire connection-state table (a finite resource, regardless of how much RAM the box has) fills up with half-open junk before any of it can time out — leaving zero room for legitimate connections to even start the handshake.

▸ this is exactly what the SYN-scan detection signature looks like, inverted

Day 09's Wireshark lesson covers spotting a port scan as many SYNs to sequential ports from one source. A SYN flood is the volumetric cousin of that same pattern — many SYNs to the same port from many (spoofed) sources, with the defining tell being SYN-ACKs that never get their final ACK. The detection logic in both cases is "count SYNs that never complete the handshake."

03

UDP amplification, packet by packet

A small lie about who's asking, turned into a flood by someone else's server doing exactly what it's supposed to.

Attacker
sends small query
spoofed src = victim IP
small
request →
Open DNS / NTP server
does exactly what it's
designed to do
large
response →
Victim
never asked for any of this,
gets flooded anyway

UDP has no handshake to verify the source address, so the attacker simply writes the victim's IP into the source field of a small query sent to an open, legitimate DNS or NTP server. That server has no way to know the source address is forged — it just sends its (much larger) response to whatever address the packet claims came from. A small outbound request from the attacker becomes a large inbound flood landing on someone who never sent anything at all, and the amplification ratio (response size ÷ request size) is exactly the leverage that makes this attack class so effective at scale.

▸ why this is specifically a UDP problem, not a DNS/NTP problem

DNS and NTP aren't uniquely broken — they're just commonly-open UDP services with naturally large response-to-request size ratios, which makes them efficient amplifiers. The actual vulnerability is structural to UDP itself: any open, unauthenticated UDP service with asymmetric response sizes is a candidate amplifier. This is also exactly why so many UDP-based protocols have since added source validation extensions or moved toward requiring authentication for non-trivial responses.

05

A correction worth making — SSH "security through port-hiding"

A genuinely common piece of advice that deserves more nuance than it usually gets.

"Change the default SSH port away from 22" shows up constantly in hardening checklists, including as a primary recommendation in some versions of this lesson. It's worth being precise about what it actually does and doesn't buy you:

What it genuinely does

  • Dramatically reduces log noise from automated, untargeted bots that only ever check port 22
  • Some real-world reports show a 90%+ drop in failed-login log entries after moving off the default port
  • Makes it easier to spot genuinely targeted activity, since the background noise disappears

What it does not do

  • Stop a determined, targeted attacker — a full 65,535-port scan takes seconds with modern tooling
  • Substitute for actual access control — a firewall already restricting source IPs makes port-hiding redundant
  • Address any underlying vulnerability in the SSH service itself
▸ the honest framing

Changing the port is security through obscurity — a legitimate, low-cost layer for reducing noise and automated-bot load, but it should never be the headline item on an SSH hardening list. The things that actually stop a real attacker are: disabling password authentication in favor of public keys, disabling root login, restricting source IPs at the firewall where feasible, and keeping the SSH daemon patched. Treat port-hiding as an optional, minor addition after those are already in place — not a substitute for any of them.

06

Hands-on protocol reconnaissance

Theory validated against real endpoints using native command-line tools.

$ Deep-dive name resolution with dig

nslookup gives you basic lookups; dig (Domain Information Groper) extracts detailed DNS zone data. Attackers sometimes abuse DNS TXT records to store command-and-control payloads or otherwise manipulate mail-security headers — see Day 09's DNS tunneling section for exactly how that encoding works.

bash
# query TXT records to audit SPF/DKIM authentication rules
$ dig google.com TXT

;; ANSWER SECTION:
google.com.      3599    IN    TXT    "v=spf1 include:_spf.google.com ~all"

That SPF record is exactly the defense named in the SMTP audit profile above — it's the receiving mail server's way of checking "is this sending IP actually authorized to send mail as google.com?"

$ Auditing web responses & headers via curl

curl -I fetches headers only, no body — fast way to check server software and whether key security headers are present.

bash
$ curl -I https://www.google.com

HTTP/2 200
content-type: text/html; charset=UTF-8
x-frame-options: SAMEORIGIN
strict-transport-security: max-age=31536000

x-frame-options: SAMEORIGIN blocks the page from being embedded in an iframe on another domain — a clickjacking defense. strict-transport-security is the HSTS header named in the HTTPS audit profile above, telling browsers to never downgrade this domain to plain HTTP again.

07

Incident responder lens

Putting today's protocol baseline to work.

Every protocol audited in Phase 04 has a predictable "normal" shape. An incident responder's actual job is recognizing the moment that shape changes: a DNS resolver suddenly answering for a domain it's never seen before (cache poisoning), a second DHCP server appearing on a segment that should only have one (rogue DHCP), a spike in half-open TCP connections with no completing ACKs (SYN flood), or an SMTP server accepting mail "From:" a domain whose SPF record explicitly says it shouldn't be sending from that IP (spoofing). None of these require exotic tooling to spot — they require knowing, cold, what each protocol's normal baseline looks like, which is exactly what today built.

09

Troubleshooting

What actually trips people up running these commands.

"dig: command not found"

dig ships as part of the dnsutils package on Debian/Ubuntu/Kali. Install with sudo apt install dnsutils -y if it's missing.

dig TXT returns nothing for a domain you know has SPF set up

Confirm you're querying the root domain, not a subdomain — SPF records typically live on the domain itself (or a dedicated _spf subdomain referenced from there), not necessarily on every subdomain. Also check whether the domain uses a custom mail provider with its own SPF include chain rather than hosting the record directly.

curl -I shows fewer headers than expected, fewer than appear in a real browser's Network tab

curl -I sends a HEAD request, and some servers respond differently to HEAD vs GET — occasionally omitting headers that only get set on the full response. Try curl -I --http1.1 to rule out HTTP/2-specific header handling differences, or fetch full headers with curl -sD - -o /dev/null <url> which performs a real GET but only prints the headers.
