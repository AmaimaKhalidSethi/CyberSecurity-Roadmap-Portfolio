The OSI Model & Network Exploration

Before you can analyze an attack, investigate an alert, or intercept credentials, you need to know exactly how data actually moves. The OSI model isn't theory for its own sake — it's the diagnostic map every later "this is a Layer 7 attack" or "Layer 3 spoofing" statement depends on.

00

Overview

The OSI model breaks network communication into 7 logical layers — and gives security professionals a shared vocabulary for describing exactly where something happened.

The real world mostly runs on the simpler 4-layer TCP/IP model, but cybersecurity professionals still reach for the 7-layer OSI model constantly, because it's more granular — "this is a Layer 7 DDoS attack" and "this is a Layer 3 packet spoofing attack" describe genuinely different problems requiring different tools, even though both technically happen "on the network." Today builds that layer-by-layer fluency, then immediately grounds it in four real command-line tools you'll use in nearly every lab from here on.

01

OSI vs. TCP/IP — the mapping nobody draws

Both models describe the same reality at different resolutions. Here's exactly how the 7 collapse into 4.

OSI Layers 5–7 (Session, Presentation, Application) collapse into a single TCP/IP Application layer, because in practice real protocols like HTTP and TLS handle session management and data formatting themselves rather than relying on distinct dedicated layers — the separation is conceptually useful but was never strictly implemented that way in real protocol stacks. OSI Layers 1–2 (Physical, Data Link) similarly collapse into TCP/IP's Link layer. Layers 3 and 4 map across cleanly 1:1 in both models — Network/Internet and Transport mean the same thing either way.

▸ why both models stay in use side by side

TCP/IP is what's actually implemented and running on every real network. OSI is what's taught and discussed, because its finer granularity gives you more precise language — "Layer 5 session hijacking" is more specific and useful in a conversation than "somewhere in the Application layer." You'll use OSI vocabulary constantly in writing and discussion, and think in TCP/IP terms when you're actually looking at a packet capture's protocol stack.

03

Testing connectivity with ping

The single fastest "is anything even there" check — operating at Layer 3.

ping uses ICMP Echo Request/Reply messages to determine if a host is alive and measure round-trip time.

bash
# send exactly 3 pings to a Google DNS server
$ ping -c 3 8.8.8.8

PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=14.2 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=117 time=15.1 ms
64 bytes from 8.8.8.8: icmp_seq=3 ttl=117 time=14.8 ms

Three fields worth reading deliberately, not just glancing past:

  • icmp_seq — sequence number, lets you spot dropped or out-of-order replies across a longer run
  • ttl — how many router hops were left when the reply arrived; this is exactly the field traceroute exploits deliberately in Phase 04
  • time — round-trip latency in milliseconds; consistently high or wildly varying values suggest congestion or an unusual routing path
▸ "no reply" doesn't always mean "host is down"

Many hosts and firewalls deliberately block or rate-limit ICMP — a non-responding ping does not reliably prove a host is offline, only that ICMP isn't getting through. This is exactly the same caveat behind Nmap's -Pn flag from Day 11 — don't treat ping silence as definitive.

04

Tracing the route — the TTL trick explained

traceroute doesn't have any special access to routing tables — it just exploits one IP header field cleverly.

bash
$ traceroute example.com

1  router.local (192.168.1.1)  1.123 ms
2  isp-gateway.net (10.14.2.1)  12.450 ms
3  core-router-ny.net (203.0.113.5)  25.671 ms

Every IP packet carries a TTL (Time To Live) field — a countdown that every router along the path decrements by 1. When a router decrements TTL to 0, it discards the packet and sends back an ICMP "Time Exceeded" message to the sender, identifying itself in the process.

  1. traceroute sends a packet with TTL = 1 The very first router on the path decrements it to 0, discards the packet, and replies with Time Exceeded — revealing hop #1's identity.
  2. Then sends a packet with TTL = 2 This one survives the first router (TTL drops to 1, still alive) but dies at the second router — revealing hop #2.
  3. Repeats with TTL = 3, 4, 5... Each successive packet survives one more hop before expiring, peeling back the path one router at a time until a packet finally reaches the actual destination and gets a normal reply instead of a Time Exceeded message.
▸ this is the same TTL field you saw in the ping output

Phase 03's ping output showed ttl=117 on the reply — that's the remaining TTL the packet had left when it reached you, after counting down from whatever value the sender started with (commonly 64, 128, or 255 depending on OS). traceroute simply runs that same mechanic in reverse and deliberately, forcing each hop to reveal itself one at a time instead of only seeing the final remaining count.

05

Checking your own interface — ip addr

Before asking what's out there, confirm what you actually are on the network — the natural companion to ping and traceroute that the basics often skip.

bash
$ ip addr

2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
    link/ether 08:00:27:4a:1b:9c brd ff:ff:ff:ff:ff:ff
    inet 192.168.56.101/24 brd 192.168.56.255 scope global eth0

ip addr (often shortened to ip a) is the modern replacement for the older ifconfig, and shows exactly the two pieces of identity that map straight onto Phase 02's layers: the link/ether MAC address (Layer 2) and the inet IP address with its subnet mask (Layer 3) — confirming both in one command before you start troubleshooting anything further up the stack.

▸ why this is the right first step, not an afterthought

If a ping or scan against a target fails, the very first thing worth ruling out is whether your own interface even has the IP/subnet you think it does — a misconfigured or missing address on your own box produces symptoms that look identical to "the target is unreachable" but have nothing to do with the target at all. Day 11's Nmap-vs-Metasploitable2 troubleshooting traced exactly this kind of mismatch back to a network adapter misconfiguration.

06

Discovering listening services with ss

The modern, faster replacement for netstat — and the fastest way to answer "what's actually listening on this box right now."

bash
# -t TCP, -u UDP, -l Listening, -p show Process, -n Numeric ports/IPs
$ ss -tulpn

Netid  State   Recv-Q  Send-Q    Local Address:Port    Peer Address:Port
tcp    LISTEN  0       128             0.0.0.0:22           0.0.0.0:* users:(("sshd",pid=742,fd=3))
tcp    LISTEN  0       511             0.0.0.0:80           0.0.0.0:* users:(("nginx",pid=815,fd=6))

What each flag actually filters

  • -t — TCP sockets only
  • -u — UDP sockets only (add alongside -t to see both)
  • -l — listening sockets only, not active connections
  • -p — show the owning process name and PID (needs root for full visibility)
  • -n — numeric output, skips slow reverse-DNS/service-name lookups

Reading the Local Address column

0.0.0.0:22 means the service listens on every interface, reachable from anywhere that can route to this host. 127.0.0.1:<port> would mean loopback-only — unreachable from the network at all, a meaningfully smaller exposure.

▸ what you're actually hunting for in this output

Two scenarios matter most: (1) a service bound to 0.0.0.0 that should only ever be reachable locally or from a specific internal subnet — an unintentionally exposed database or admin panel; (2) a process name or port you simply don't recognize at all — every line in this output should be explainable. An unfamiliar PID holding an open listening socket is one of the more reliable signs of an unauthorized backdoor or implant, precisely because malware has to listen somewhere if it wants to receive commands.

If you're monitoring a honeypot, running ss -tulpn confirms your decoy services are correctly bound and listening for attacker traffic before you trust any alert it generates — an alert from a service that was never actually listening is a configuration bug, not a real detection.

07

Blue team reading of the same four tools

Tying today's commands directly to the OSI layers they each operate at.

ToolOSI layerBlue-team question it answers
ping3 — NetworkIs this host reachable at all, and how's the latency?
traceroute3 — NetworkWhat's the actual path, and where along it does something go wrong?
ip addr2 / 3 — Data Link / NetworkWhat am I, identity-wise, on this network right now?
ss -tulpn4 — TransportWhat's actually listening on this box, and is all of it explainable?
▸ the diagnostic order this naturally suggests

When something's wrong, work bottom-up exactly like the OSI model itself: confirm your own interface first (ip addr), then reachability (ping), then the path if reachability fails (traceroute), then — once you're investigating a specific host rather than connectivity in general — what's actually listening there (ss). This order isn't arbitrary; it mirrors which layer each tool operates at, lowest first.

09

Troubleshooting

What actually goes wrong running these four commands.

ping shows "Destination Host Unreachable" or 100% packet loss

Check ip addr first (Phase 05) — confirm your own interface has a valid IP and is actually up. If your own config looks correct, the target may simply be blocking ICMP rather than being genuinely offline; see the warning callout in Phase 03.

traceroute: command not found

On Debian/Ubuntu/Kali: sudo apt install traceroute -y. Some minimal containers and distros ship only tracepath or expect mtr instead — both achieve a similar result with slightly different output formatting.

traceroute output shows several lines of "* * *" instead of hop details

Expected and common — some routers along the path are configured not to send back ICMP Time Exceeded replies, or rate-limit them so heavily that traceroute's default timeout gives up before a reply arrives. This doesn't mean the path is broken, only that those specific hops chose not to identify themselves; the trace typically continues past them to later hops that do reply.

ss -tulpn doesn't show the process name/PID, just a dash

Process name resolution for -p requires root privileges to read other users' process information — re-run with sudo ss -tulpn if you're not already root.
