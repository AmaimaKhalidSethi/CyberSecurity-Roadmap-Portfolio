Overview

A firewall is a trust-boundary enforcement mechanism, not a single product — three genuinely different technologies share the name.

A firewall enforces trust boundaries by controlling the ingress and egress traffic profile of a system or network, examining traffic against explicit, predefined rules. The word "firewall" covers three technically distinct generations of mechanism, and which generation you're dealing with changes what it can see, what it can be tricked by, and how it should be configured. Today builds that architecture up layer by layer, then grounds it in the actual ports you'll see in every scan you ever run, and finishes with hands-on UFW configuration on a real host.

01

The single fact that explains every capability difference between the three generations below: how high up the stack each one actually looks.

Every firewall generation reads Layers 3 and 4 — source/destination IP and port. The entire evolution story in Phases 02–04 is about what gets added on top: first memory of connection state, then visibility all the way up to Layer 7 where the actual application content lives. Keep this picture in mind — it's the one mental model that makes "why is NGFW harder to evade than stateful" click immediately.

02

Stateless packet filtering (first generation)

Layers 3–4 only, and — critically — no memory between packets.

Stateless filters analyze each packet in complete isolation, with no record of what came before. Each packet is judged purely on its own header fields:

  • Inspection scope: source IP, destination IP, source port, destination port, protocol type (TCP/UDP/ICMP)
  • The structural vulnerability: because there's no connection tracking, the filter cannot distinguish a legitimate reply to a request your own network made from an unsolicited packet crafted to look like one
▸ the asymmetric-rule problem, concretely

For an internal client to download a webpage, an engineer using a stateless filter has to write a static rule permitting inbound traffic from source port 80/443 back into the internal segment — because the filter has no way to know "this inbound packet is a reply to something we asked for" versus "this inbound packet is unsolicited." That static rule is now permanently open, whether or not your network actually has an active request in flight. An attacker can craft a raw packet with source port 80 and an ACK flag set, and a stateless filter — having no session memory to check it against — will wave it straight through, believing it's a legitimate reply.

03

Stateful inspection (second generation)

Same Layers 3–4 visibility, but now with memory of the conversation.

Stateful inspection bridges the contextual gap stateless filtering leaves open by actively monitoring the lifecycle of each network transaction, using an internal tracking structure usually called the state table (or connection tracking table — conntrack on Linux specifically).

  1. Outbound request initiates a tracked entry When an internal host sends a TCP SYN, the firewall records the connection's full tuple — source/destination IP, source/destination port, and current TCP state (SYN-SENT, ESTABLISHED, etc.) — into the state table.
  2. Inbound reply is matched against that entry, not a static rule When the external host replies, the firewall checks whether the reply's tuple matches an entry already marked ESTABLISHED or RELATED in the table. If it matches, the packet is allowed through automatically — no static "allow port 80 inbound" rule is needed at all.
  3. Unsolicited traffic has nothing to match against An attacker's spoofed ACK packet, or a probe against a port nobody asked anything from, finds no corresponding entry in the state table and gets dropped — solving exactly the asymmetric-rule problem from Phase 02, without leaving permanently-open static rules.
▸ this is what your home router already does

Stateful inspection is the default behavior of essentially every consumer router and Linux's built-in netfilter/iptables framework (which UFW configures in Phase 09) — it's why you can browse the web from behind a home router with zero manual port-forwarding rules, but a port scan from outside still finds everything closed/filtered.

04

Next-generation firewalls (Layer 7 aware)

Stateful inspection knows who's talking to whom. NGFW asks what they're actually saying.

Adversaries adapted to stateful boundaries by tunneling malicious traffic — reverse shells, C2 check-ins — over the ports that are always open by necessity: 80 and 443. A stateful firewall sees "established TCP connection to port 443" and allows it, because that's indistinguishable at Layers 3–4 from a person checking their email. NGFWs close this gap with Deep Packet Inspection (DPI), reading further up the stack into Layer 7 — the actual application content.

Application identification (App-ID)

Looks at payload behavior patterns — packet timing, TLS handshake fingerprints (e.g. JA3-style client fingerprinting), and protocol-specific signatures — to identify what application is actually generating the traffic, regardless of which port it's running over.

User identity context (User-ID)

Integrates with directory services like Active Directory to bind policy to a real user/group identity rather than a DHCP-assigned IP that could belong to anyone at any given moment.

TLS/SSL interception

The firewall acts as a man-in-the-middle by design: it terminates the client's TLS connection, inspects the plaintext, then re-encrypts before forwarding to the real destination — substituting its own certificate, trusted because it's pushed to managed endpoints in advance.

Inline threat prevention

Scans decrypted content for known exploit signatures and anomalous behavior before the data ever reaches the destination host — closer to an IDS/IPS folded directly into the firewall path.

▸ a precision correction worth making explicitly

TLS interception at scale is genuinely demanding — high-end appliance NGFWs often use dedicated ASICs or hardware crypto accelerators to do it without crushing throughput, but that's a performance detail of certain hardware tiers, not a defining requirement of NGFW technology itself. Plenty of NGFW deployments (including most software/cloud-based ones) do TLS inspection entirely in software. Don't treat "ASIC-accelerated" as a universal NGFW property — it's an implementation choice some vendors make at certain price points.

How App-ID actually catches something like Tor hidden inside port 443: Tor's TLS handshake has observable fingerprint characteristics (certificate patterns, cipher suite ordering, traffic timing) that differ subtly from ordinary browser-to-website TLS, even though both are "port 443, encrypted." NGFW signature databases are built specifically to catalog these distinguishing fingerprints across thousands of known applications and tunneling tools.

Why each generation exists — the arms race in one table

Every generation exists because attackers found the blind spot in the previous one.

GenerationWhat it seesThe gap that forced the next generation
StatelessIP + port, per packet, no memoryCan't distinguish a real reply from a spoofed one — forces permanently-open static rules
StatefulIP + port + connection lifecycleCan't see inside an allowed, established connection — attackers tunnel C2 traffic through ports 80/443
NGFW (Layer 7)Actual application identity inside the allowed connectionStill has to keep up with an ever-growing signature database and new evasion fingerprints — an arms race that never fully closes
▸ no generation makes the previous one obsolete

A real-world NGFW deployment still does stateful inspection underneath — Layer 7 awareness is additive, not a replacement. When you configure UFW in Phase 09 today, you're working at the stateful generation; that's the right and sufficient layer for a single host, while NGFW logic typically lives at the network perimeter, not on every endpoint.

Core network ports & service risk

A handful of ports account for a disproportionate share of real-world findings — know these cold.

PortProtoServiceRiskSecurity risk & exploitation context
21TCPFTPHighCredentials and data sent in cleartext — passively sniffable. Susceptible to brute-force and anonymous-access misconfiguration.
22TCPSSHMediumEncrypted by design, but heavily targeted by global brute-force scanning. Risk shifts to weak ciphers, password auth left enabled, and private key leakage/reuse.
23TCPTelnetCriticalEntirely cleartext remote shell — credentials and full session content readable from any packet capture on the path. No legitimate reason to run this in 2026 outside legacy embedded gear.
25TCPSMTPMediumOpen relay misconfiguration enables spam abuse at scale. Legacy commands like VRFY can enable username enumeration against the mail server.
53UDP/TCPDNSHighUDP for queries, TCP for zone transfers. Enables DNS amplification DDoS (see Phase 06 callout below), cache poisoning, and is a common covert C2/data-exfiltration channel via DNS tunneling.
80TCPHTTPHighUnencrypted — traffic interception, session token theft, and the full surface of web app vulnerabilities (SQLi, XSS, command injection) layer on top of this transport.
443TCPHTTPSMediumEncrypted transport, but precisely because it's encrypted and always allowed outbound, it's the favored channel for C2 callbacks and exfiltration — see the NGFW discussion in Phase 04.
445TCPSMBCriticalWindows file/print sharing. History of severe remote-code-execution flaws (EternalBlue/MS17-010 — see callout). Should essentially never be reachable from outside a trusted internal segment.
3389TCPRDPCriticalWindows remote desktop. Among the most common initial-access vectors for ransomware — credential spraying against internet-exposed RDP, and historical flaws like BlueKeep (see callout).
▸ how DNS amplification actually works, in one paragraph

An attacker sends a small DNS query to an open resolver, but spoofs the source IP to be the victim's address. The resolver — doing exactly what DNS resolvers are supposed to do — sends its (much larger) response to the spoofed address, not back to the attacker. A small request becomes a large flood landing on someone who never asked for it, and the attacker's own IP never appears anywhere in the traffic the victim sees. This single mechanism is why "open DNS resolver" is treated as a finding worth fixing even when nothing else about the DNS config looks wrong.

07

Scan mechanics — Connect vs SYN, packet by packet

Port scanning is the analytical practice of probing a target to discover live services — here's exactly what's on the wire for the two foundational techniques.

-sT — Full TCP Connect scan

Completes the textbook three-way handshake on every probed port:

  • Attacker → SYN
  • Target → SYN-ACK (port open) or RST (closed)
  • Attacker → ACK, then immediately RST/FIN to tear down

Requires no special privileges — it's just the OS's normal socket connect() call. But because a full connection is established, it's logged by the target's normal application-layer connection logging, every time.

-sS — SYN / half-open scan

Deliberately interrupts the handshake before a full session registers:

  • Attacker → SYN
  • Target → SYN-ACK (open) or RST (closed)
  • Attacker → RST immediately (open case) — never sends the final ACK

Requires root/admin privileges, since crafting a raw packet that deviates from the OS's normal TCP state machine needs raw socket access. Many applications never see this at the connection-log level since no session was ever fully established.

bash
# stealth SYN scan with version detection against common services
$ sudo nmap -sS -sV 192.168.1.100

# evaluate every one of the 65,535 ports — slow, thorough
$ sudo nmap -p- 192.168.1.100

# aggressive sweep: OS fingerprint, NSE default scripts, traceroute
$ sudo nmap -A 192.168.1.100
▸ "stealthier" is a relative, not absolute, claim

A SYN scan avoids application-layer connection logs because no full session forms. It does not avoid detection at the network layer — any firewall or IDS watching raw packet flow (the stateful/NGFW devices from Phases 03–04) sees the exact same pattern either way: rapid SYNs across many ports with no legitimate follow-through. The "stealth" in SYN scan refers to a specific, narrower blind spot, not invisibility.

08

Port state classifications

Three words, three completely different stories about what's actually happening on the target.

  • Open — an active process is listening and responded affirmatively. This is real, present attack surface, and the starting point for any vulnerability assessment of that service.
  • Closed — the target returned a definitive RST. The host is reachable and the network path works, but nothing is bound to that specific port right now.
  • Filtered — Nmap's probe was silently dropped by an intermediate device, with no RST and no SYN-ACK returned. You genuinely don't know if a service is there; the firewall is hiding the answer, not the host itself.
▸ filtered is a question, not an answer

Closed tells you something definite. Filtered tells you a firewall is in the way and nothing more — it's worth a second probe (different scan type, different timing, a different source port) before you conclude there's nothing of interest behind it.

09

Defensive engineering — host-based firewall control with UFW

Network perimeter defenses are insufficient once an attacker is already inside a subnet via phishing or an edge exploit — host-level rules stop lateral movement.

UFW (Uncomplicated Firewall) is a friendlier management layer over Linux's kernel-level netfilter framework — the same stateful packet-filtering engine described in Phase 03, just running on a single host instead of at a network perimeter.

① Establish a zero-trust baseline

A secure default starts from deny-everything-inbound, then explicitly allows only what's needed.

bash
# check the current state before changing anything
$ sudo ufw status verbose

# set the global default posture
$ sudo ufw default deny incoming
$ sudo ufw default allow outgoing
▸ set your SSH allow rule before enabling — every time

If you're connected over SSH and run ufw default deny incoming followed immediately by ufw enable without first allowing your SSH port, you will lock yourself out of a remote box the instant the rule applies. Always add the SSH allow rule (Step 2 below) before running ufw enable, and double-check it on a remote host before disconnecting.

② Configure safe administration and application paths

bash
# explicit SSH allow — do this before enabling, see warning above
$ sudo ufw allow 22/tcp

# permit standard web service ports via named application profiles
$ sudo ufw allow http
$ sudo ufw allow https

③ Granular network restrictions

Hardening sensitive backend services means restricting access to specific known sources, not opening them to any address.

bash
# SSH only from one designated administration workstation
$ sudo ufw allow from 203.0.113.42 to any port 22 proto tcp

# MySQL reachable only from a specific internal subnet
$ sudo ufw allow from 192.168.10.0/24 to any port 3306 proto tcp

# explicitly drop a known-hostile source
$ sudo ufw deny from 198.51.100.77 to any

④ Activate and audit

bash
# turn it on — persists across reboots automatically
$ sudo ufw enable

# review the active rule set with numbered indices, for easy deletion later
$ sudo ufw status numbered

10

SOC & architecture perspectives

The same firewall data, read for two different jobs.

SOC Analyst / Threat Hunter

Watch for port-misuse anomalies — a protocol running on a non-standard port for it (SSH traffic appearing inside what should be port-80 flows, for instance). Rapid sequential drop events from a single external IP against many destination ports is the classic signature of an active port scan in progress. Sudden bursts of outbound connection attempts being dropped from an internal server can indicate a compromised host trying to reach a C2 node that's since been blocked or doesn't exist.

Security Architecture / Engineering

Implement micro-segmentation rather than relying solely on perimeter defense — host-level rules (UFW, from Phase 09) separate endpoint ranges, staging environments, and core databases from each other, so a single compromised host doesn't grant lateral access to everything. Run continuous, scheduled Nmap sweeps against your own internal and external ranges — finding an unexpected open port during a routine internal audit is dramatically cheaper than finding it after an incident.

12

Troubleshooting UFW

The mistakes that actually happen during this exact lab.

Locked out of a remote VM immediately after running ufw enable

This is the single most common UFW mistake — see the danger callout in Phase 09. If you still have console/hypervisor access (e.g. the VirtualBox GUI console, not SSH), log in directly and run sudo ufw allow 22/tcp then sudo ufw reload. If you have no console access at all, you may need to use your cloud provider's serial/recovery console to fix the rule remotely.

ufw status shows "inactive" even after running ufw enable

Confirm you actually ran sudo ufw enable and answered "y" to the confirmation prompt — it's easy to run the command, get the disruption warning, and not realize the prompt needs a response. Re-run sudo ufw status verbose immediately after to confirm state actually changed.

Rules for an IP range don't seem to apply as expected

Check rule order with sudo ufw status numbered — UFW (like the underlying netfilter rules) evaluates top to bottom and stops at the first match. A broad allow rule placed above a more specific deny rule for the same traffic will win, even if the deny rule looks more specific on paper.

IPv6 traffic seems to bypass rules written for an IPv4 address

UFW rules targeting a specific IPv4 address or range have no effect on IPv6 traffic — they're entirely separate rule sets. If the host has IPv6 enabled and reachable, you need a parallel rule for the IPv6 equivalent, or disable IPv6 entirely if it's not in use, per the DigitalOcean reference in Phase 09.
