HTTP, HTTPS, Requests, Responses & Cookies — in-depth guide

Every web vulnerability you'll learn from here on — SQL injection, XSS, CSRF, session hijacking, auth bypass — happens inside an HTTP request or response. Before touching Burp Suite, you need to be fluent in reading the conversation it's intercepting.

00

Overview & why this matters

The protocol underneath every login page, banking site, and API call you'll ever test.

HTTP (HyperText Transfer Protocol) is how web browsers and web servers talk to each other. Every time you open a website, your browser sends a request, and the server sends back a response. That exchange — and nothing more exotic than that — is the substrate that every web attack technique you'll study later sits on top of.

▸ the restaurant analogy

You place an order (Request) → the kitchen prepares it → the food arrives at your table (Response). Every HTTP exchange follows this exact shape, no matter how complex the website. Keep this mental model — it makes headers, cookies, and status codes click much faster than memorizing them as isolated facts.

As a SOC Analyst, Security Analyst, Penetration Tester, or Bug Bounty Hunter, you will spend a large fraction of your working hours staring at exactly this: requests, responses, cookies, headers, and tokens, trying to figure out what happened or what can be made to happen differently.

01

What is HTTP?

The simplest possible example, traced end to end.

You type a URL. Three things happen, in order:

conceptual trace
# 1. you type this into the address bar
https://example.com

# 2. browser sends this request
GET / HTTP/1.1

# 3. server replies with this status line, plus the page itself
HTTP/1.1 200 OK

That's the entire core loop. Everything else in this guide — methods, headers, cookies, status codes — is detail layered on top of "browser asks, server answers."

02

HTTP vs HTTPS

Same protocol, one critical difference: whether anyone watching the wire can read what's on it.

⚠ HTTP — http://example.com

  • Data sent in plain text
  • Anyone intercepting traffic can read it
  • Uses port 80
  • No confidentiality, integrity, or authentication guarantees
VS

✓ HTTPS — https://example.com

  • HTTP + TLS encryption
  • Encrypts all communication between browser and server
  • Uses port 443
  • Provides confidentiality, integrity, and authentication

Most modern websites use HTTPS by default now — but as a security learner you'll deliberately spend time in HTTP-only lab environments (like DVWA over plain HTTP) precisely because you can see every byte of the traffic without decryption getting in the way first.

▸ TLS is doing the heavy lifting

HTTPS isn't a different protocol from HTTP so much as HTTP running inside an encrypted TLS tunnel. The request/response structure you learn today is identical either way — TLS just wraps it in a layer the browser and server negotiate before any HTTP data is exchanged.

03

The request/response flow

Every webpage you've ever loaded followed exactly these five steps.

  1. Browser sends a Request Built from the URL you typed, a link you clicked, or a form you submitted.
  2. Server receives the Request The web server (Apache, Nginx, IIS, etc.) accepts the connection and parses what was asked for.
  3. Server processes it Runs application logic — queries a database, checks authentication, builds the page — whatever the requested resource requires.
  4. Server sends a Response Status code, headers, and (usually) a body containing HTML, JSON, an image, or other content.
  5. Browser displays the page Parses the HTML/CSS/JS in the response body and renders what you see on screen.
▸ why this flow is the foundation for everything later

SQL injection happens at step 3 (malicious input changes what the server's processing logic does). XSS happens at step 5 (malicious content in the response gets executed by the browser instead of just displayed). Session hijacking exploits step 1/2 (stealing or forging the credentials that prove who's making the request). You cannot meaningfully understand any of those attacks without this five-step model first.

04

Anatomy of an HTTP request

What the client actually sends, byte for byte.

A request is built from four parts:

  • Request Method — what kind of action (GET, POST, etc.)
  • URL — which resource on the server
  • Headers — metadata about the request
  • Cookies — small stored values sent along automatically
  • Body — data payload (present on POST/PUT, usually absent on GET)
raw HTTP request
GET /login HTTP/1.1
Host: example.com
User-Agent: Firefox
Accept: text/html

Line 1 is the request line — method, path, and HTTP version. Every line after that is a header — a Name: value pair. This particular example has no body because it's a GET request asking the server to hand over the login page, not submitting anything to it.

05

HTTP methods

What kind of action the request is asking the server to perform.

GET

Requests data. Used for viewing pages, searching, reading data.

  • GET /products

POST

Sends data to the server. Used for login, registration, forms, uploads.

  • POST /login

PUT

Updates existing data — typically replaces a resource in full.

DELETE

Deletes existing data on the server.

PATCH

Updates part of an existing resource, rather than replacing it entirely like PUT.

▸ why GET vs POST matters for security

GET requests put their parameters in the URL itself — visible in browser history, server logs, and the address bar. POST requests put parameters in the body, slightly less exposed but not encrypted or hidden by default — over plain HTTP, a POST body is just as readable to an interceptor as a GET parameter. This distinction matters a lot once you start testing for parameter tampering and injection points.

06

Anatomy of an HTTP response

What comes back after the server processes your request.

A response contains three parts:

  • Status Code — did it work, and if not, why not
  • Headers — metadata about the response
  • Response Body — the actual content (HTML, JSON, image bytes, etc.)
raw HTTP response
HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>

Line 1 is the status line — HTTP version, status code, and a human-readable reason phrase ("OK"). Everything after is headers, then a blank line, then the body.

07

HTTP status codes

The single fastest signal for what just happened to a request — learn these cold.

CodeClassMeaning
200 OKSuccessEverything worked successfully.
301 / 302RedirectPage redirects somewhere else.
400 Bad RequestClient errorClient sent an invalid request.
401 UnauthorizedClient errorAuthentication required.
403 ForbiddenClient errorServer understands the request but refuses access.
404 Not FoundClient errorRequested page doesn't exist.
500 Internal Server ErrorServer errorProblem occurred on the server.
▸ the quick mental grouping

2xx = it worked. 3xx = go look somewhere else. 4xx = you (the client) did something the server didn't like. 5xx = the server broke on its own end, not because of anything you sent. This grouping alone will get you most of the way through reading unfamiliar status codes you haven't memorized yet.

08

Headers

Metadata riding alongside every request and response — and a major target during security testing.

Common request headers

  • Host
  • User-Agent
  • Cookie
  • Accept
  • Authorization

Common response headers

  • Content-Type
  • Content-Length
  • Server
  • Set-Cookie
  • Cache-Control

During security testing, headers are where you'll find authentication tokens (Authorization), session identifiers (Cookie / Set-Cookie), server fingerprinting clues (Server), and content-handling behavior (Content-Type) that can reveal how the application processes input.

09

Cookies

Small pieces of state the server asks your browser to hold onto and send back.

Cookies are small pieces of data stored in the browser, used so a website can remember who you are between requests — since HTTP itself has no built-in memory of previous requests.

example cookie
Cookie: PHPSESSID=abc123xyz

Cookies are commonly used for:

  • Login sessions
  • User preferences
  • Shopping carts
  • Tracking

◷ Types of cookies

Session Cookie Deleted when the browser closes — exists only for the current browsing session.
Persistent Cookie Stored until a set expiration date, surviving across browser restarts.
Secure Cookie Sent only over HTTPS — never transmitted over an unencrypted connection.
HttpOnly Cookie Cannot be accessed using JavaScript — helps reduce the impact of XSS attacks by keeping the cookie out of reach of injected scripts.
▸ Secure and HttpOnly are flags, not cookie types

Technically Secure and HttpOnly are attributes set on a Set-Cookie header (Set-Cookie: PHPSESSID=abc123; HttpOnly; Secure), not separate kinds of cookies from Session/Persistent. A single cookie can be persistent and Secure and HttpOnly all at once. Worth keeping straight once you start reading real Set-Cookie headers in DevTools.

10

Sessions

How a stateless protocol fakes having a memory.

A session allows a website to remember who you are after logging in. Instead of asking for your password on every single page load, the server generates a unique session ID, stores it server-side, and gives your browser a copy as a cookie. Every subsequent request includes that cookie, and the server uses it to look up who you are.

example session cookie
Cookie: PHPSESSID=8f2c9d1a
▸ the core risk in one sentence

If an attacker steals this cookie — through XSS, network sniffing on unencrypted HTTP, or a misconfigured CORS policy — they can present it to the server themselves and be treated as you, fully authenticated, without ever knowing your password. This is session hijacking, and it's why cookie handling gets so much scrutiny in security testing.

11

Why cookies matter in cybersecurity

The attack categories that all trace back to today's material.

Session Hijacking

Stealing or guessing a valid session ID to impersonate a logged-in user without their credentials.

Cookie Theft

Extracting cookie values via XSS, malware, packet sniffing on unencrypted connections, or a misconfigured app exposing them.

Session Fixation

Forcing a victim to use a session ID the attacker already knows, then waiting for them to authenticate under it.

Cross-Site Scripting (XSS)

Injecting script into a page that, when it runs in another user's browser, can read and exfiltrate their cookies — unless HttpOnly blocks it.

Protecting cookies — through HttpOnly, Secure, proper expiration, and session regeneration after login — is one of the highest-leverage, lowest-effort wins in web application security. You'll see all four of these attack categories again, in depth, later in the roadmap.

12

Hands-on lab — DVWA, DevTools & curl

2–3 hours. Stop reading about HTTP and go watch it happen.

  1. Open DVWA Start Apache and MySQL if needed, then navigate to:
    browser address bar
    http://127.0.0.1/DVWA
    Log in with username admin, password password.
  2. Open browser Developer Tools Press F12, go to the Network tab, clear previous requests, then refresh the page. Watch requests appear in real time as the page loads.
  3. Inspect an HTTP request Click any request in the list and observe its Method, URL, Headers, Cookies, and Response tabs. Write down what you find for at least one request — this written observation is part of today's deliverable.
  4. Explore cookies Open the Application tab (Chrome) or Storage tab (Firefox), find Cookies, and identify PHPSESSID, the DVWA security-level cookie, and any others present.
  5. Use curl for a head-only request
    bash
    $ curl -I http://127.0.0.1/DVWA

HTTP/1.1 200 OK
Date: Sat, 27 Jun 2026 10:15:02 GMT
Server: Apache/2.4.x
Set-Cookie: PHPSESSID=abc123; path=/
Content-Type: text/html; charset=UTF-8
    Observe the HTTP version, status code, Server header, and Content-Type.
  6. Use curl verbose mode for the full exchange
    bash
    $ curl -v http://127.0.0.1/DVWA
    -v shows the complete request and response, including the TCP connection setup — this is the closest the command line gets to what DevTools shows you visually.
  7. Compare HTTP and HTTPS directly
    bash
    $ curl -I https://google.com
$ curl -I http://example.com
    Notice that the HTTPS request negotiated TLS before any HTTP data was exchanged — visible if you add -v to see the TLS handshake lines ahead of the HTTP status line.
  8. Observe a real login request Log in to DVWA again with Network tab open. Find the login POST request and identify its Method, URL, Response Code, and any cookies created in the response (Set-Cookie header) versus what was sent in the request (Cookie header).
▸ if curl isn't installed
$ sudo apt install curl -y

Beginner tips & troubleshooting

What actually trips people up on Day 14, and the daily habit that compounds.

Headers feel overwhelming — there are so many of them

Don't try to memorize every HTTP header today. Focus on understanding the purpose of requests, responses, cookies, and status codes first — the long tail of specific header names sinks in naturally once you've inspected enough real traffic.

Not sure where to look in the browser

Use Developer Tools constantly — they are one of the single most valuable tools in a web security career. Practice inspecting different websites (not just DVWA) to see how requests and responses vary across real-world apps.

curl: command not found

Install it: sudo apt install curl -y

▸ daily practice habit

Spend 10–15 minutes each day opening your browser's Network tab on a site you'd visit anyway, and try to identify the HTTP method, response status code, cookies, and important headers without looking anything up. Regular low-effort repetition like this is what makes HTTP feel automatic before you move into Burp Suite, where you'll be reading dozens of these per minute.

Keep today's notes — they get reused heavily in upcoming lessons on Burp Suite, SQL Injection, XSS, authentication, and session management.
