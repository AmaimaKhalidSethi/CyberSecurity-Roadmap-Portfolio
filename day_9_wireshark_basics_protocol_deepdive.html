Networking Fundamentals III – Wireshark Basics

An In-Depth Blueprint for Packet Analysis, Display Filters, and Protocol Inspection

1. Foundations of Packet Analysis & OSI Encapsulation

Before navigating Wireshark's interface, it is critical to understand how data moves across a network. Data transmission relies on encapsulation. When an application generates data, it passes down the Open Systems Interconnection (OSI) model stack, with each layer wrapping the upper-layer data with its own header controls.

In Wireshark, this mapping is visualized directly in the Packet Details Pane. Each expansion block represents a specific layer of the network stack, structured outward from physical to application layers:

Ethernet Header (L2)
MAC Addresses
IP Header (L3)
Source/Dest IPs & TTL
TCP/UDP Header (L4)
Ports, Sequence Nos.
Application Data (L5-L7)
DNS, HTTP, TLS Payload

When troubleshooting or threat hunting, a security analyst strips away these layers (decapsulation) inside Wireshark to isolate anomalies, pinpoint communication failures, or extract signs of malicious compromise.

2. Mastering Wireshark Display Filters

Live networks process millions of packets per minute. Finding the specific indicators of compromise (IoCs) requires mastering Wireshark's filtering syntax. Display filters do not discard packets; they hide irrelevant data from view.

Logical Operators

  • and / && : Both conditions must be met (e.g., ip.addr == 10.0.0.5 && tcp.port == 80)
  • or / || : Either condition can be met (e.g., http || dns)
  • not / ! : Excludes conditions (e.g., !arp && !ssdp)

Essential Threat Hunting Display Filters

Isolating Specific Host Communications

To view all traffic associated with a suspicious IP address, regardless of whether it is the sender or receiver:

ip.addr == 192.168.1.50

To isolate directional traffic (e.g., assessing an outbound data exfiltration attempt):

ip.src == 192.168.1.50 and ip.dst == 203.0.113.5

Filtering by Layer 4 Ports

Isolate specific services such as non-standard HTTP configurations or active connections to known ports:

tcp.port == 80 or tcp.port == 443

To search for traffic targeting a specific destination port that might indicate reconnaissance scanning:

tcp.dstport == 22

Finding Specific Content and String Matching

Locate specific text patterns across unencrypted packets (e.g., hunting for an exploit string, specific user-agent, or unauthorized commands):

frame contains "password"

Using regular expressions to isolate variations of strings across payloads:

matches "select.*from"

3. Protocol Deep Dive: What to Search & How to Analyze

Investigating network activity requires knowing exactly what normal communication looks like versus how adversaries abuse protocols. Below is a detailed breakdown of how to isolate, inspect, and analyze core network protocols.

A. Domain Name System (DNS)

What to Search: Type dns into the display filter box. To isolate malicious patterns like DNS tunneling or rapid queries to C2 (Command & Control) channels, use:

dns.flags.response == 0 // Shows queries only
dns.flags.response == 1 // Shows answers only
dns.qry.type == 16 // Filters for TXT records (often used in data exfiltration)

How to Find and Understand the Fields:

  1. Apply the filter dns and select a packet.
  2. Expand the Domain Name System layer in the Packet Details Pane.
  3. Inspect the Transaction ID: Queries and their matching responses share identical Transaction IDs. If you see responses without matching queries, suspect spoofing.
  4. Expand the Queries sub-tree: Look at dns.qry.name. Attackers often bypass firewalls by encoding malicious payloads into subdomains (e.g., afe394b92c.attacker.com). Anomalously long domain strings indicate potential DNS Tunneling.
  5. Expand the Answers sub-tree: Inspect the time-to-live (TTL) fields. Malicious domains tied to Fast-Flux networks change rapidly and will feature ultra-low TTL values (e.g., less than 60 seconds).

B. Transmission Control Protocol (TCP) & The Three-Way Handshake

What to Search: To analyze the underlying reliability of network conversations or detect port scanning, use:

tcp.flags.syn == 1 and tcp.flags.ack == 0 // Isolates Handshake connection requests
tcp.flags.reset == 1 // Filters aborted connections (RST flags)
tcp.analysis.retransmission // Flags high network latency or packet drops

How to Find and Understand the Fields:

  1. The Three-Way Handshake: Look for three successive packets matching a conversation.
    • Packet 1: [SYN] Sent by client. Look inside Flags for Syn: Set. Sequence Number is 0 (relative).
    • Packet 2: [SYN, ACK] Sent by server. Look for both Syn: Set and Acknowledgment: Set.
    • Packet 3: [ACK] Sent by client. Look for Acknowledgment: Set. The connection is now established.
  2. Anatomy of a Port Scan: If you apply tcp.flags.syn == 1 and observe a single external IP sending thousands of SYN packets to sequential destination ports within seconds, followed immediately by RST (Reset) packets from your server, you are witnessing an active network reconnaissance scan.
  3. Following Streams: Right-click any TCP packet, navigate to Follow → TCP Stream. This stitches fragmented packets into a chronological, human-readable transcript of the data exchange.

C. Hypertext Transfer Protocol (HTTP)

What to Search: Because cleartext HTTP traffic is highly transparent, it is an ideal hunting ground for uncovering malware binaries, unencrypted credentials, and automated attacks.

http.request.method == "POST" // Isolates data transmissions (logins, uploads)
http.response.code == 404 // Finds file-not-found errors (web scanning indicators)
http.user_agent matches "sqlmap" // Catches automated scanning tool signatures

How to Find and Understand the Fields:

  1. Filter by http. In the packet list view, look for the request method (GET/POST) and server responses (200 OK, 302 Redirect).
  2. Expand the Hypertext Transfer Protocol layer.
  3. Examine the Host and User-Agent fields. Legitimate browsers have complex User-Agent strings. Web shells or malicious download scripts often use generic or empty User-Agents (e.g., Wget/version or curl/version).
  4. Inspect the Full request URI: Check for directory traversal attempts (e.g., ../../etc/passwd) or embedded SQL injections within the URL string parameters.

D. Hypertext Transfer Protocol Secure (HTTPS / TLS)

What to Search: Modern production environments encrypt traffic via Transport Layer Security (TLS). While you cannot read the encrypted payload directly without certificate keys, you can analyze the handshake handshake attributes to fingerprint malicious servers.

tls.handshake.type == 1 // Filters for Client Hello packets
tls.handshake.type == 2 // Filters for Server Hello packets
tls.handshake.extensions_server_name // Extracts the actual SNI domain requested

How to Find and Understand the Fields:

  1. Filter by tls or ssl.
  2. Locate the Client Hello packet. Expand the TLS layer → Handshake Protocol → Extensions. Locate the server_name (SNI) field. This shows the actual domain the user is connecting to before encryption conceals the session. Compare this to DNS records to find anomalies.
  3. Locate the Server Hello packet. Expand the certificates section. Inspect the Issuer and Validity Period. Self-signed certificates, certificates issued by unknown authorities, or validities lasting only a few days are highly characteristic of temporary Command & Control infrastructure.

4. Investigating Network Activity: SOC Analyst Playbook

When an alert fires in a Security Operations Center (SOC), a packet capture (PCAP) provides the ground truth. Follow this sequential operational methodology to isolate security events:

Step 1: Check High-Level Statistics

Before writing single filters, get a global view of the data. Navigate to Statistics → Conversations. Click on the TCP or IPv4 tabs to sort by bytes transferred. A single local host uploading gigabytes of data to an external, unfamiliar IP indicates data exfiltration.

Step 2: Profile Protocol Distribution

Navigate to Statistics → Protocol Hierarchy. This window breaks down the percentage of traffic per protocol. If an unprivileged zone (like a Guest Wi-Fi network) shows 80% of its traffic allocated to protocol layers like SSH, RDP, or SMB, immediate lateral movement investigations should be launched.

Step 3: Extract Artifacts and Payloads

If you discover a malicious file download over unencrypted HTTP, you can reconstruct the binary directly from the packet stream. Navigate to File → Export Objects → HTTP.... Review the filenames, select the threat item, and save it locally to a sandbox environment for static malware analysis and hashing.

Pro Tip for Threat Hunters: Always correlate your Wireshark analysis with host-based logs (Sysmon, EDR event tracking). Finding a suspicious HTTP POST connection to a rogue IP in Wireshark becomes significantly more critical when host logs reveal the connection was spawned directly from a malicious macro running inside Microsoft Word!
