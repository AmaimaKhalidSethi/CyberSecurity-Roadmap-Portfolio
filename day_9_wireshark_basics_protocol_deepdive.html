Wireshark Basics & Protocol Deep Dive

A packet capture is ground truth — it doesn't lie the way logs sometimes do, because it's the traffic itself. Today is about reading that ground truth fluently: encapsulation, display filter syntax, and what normal vs. malicious actually looks like across DNS, TCP, HTTP, and TLS.

00

Overview

Before navigating Wireshark's interface, you need the mental model underneath it: encapsulation.

Data transmission relies on encapsulation. When an application generates data, it passes down the OSI model stack, and each layer wraps the data handed to it from above with its own header. Wireshark's Packet Details Pane visualizes this directly — each expandable block in that pane corresponds to one layer of the stack, ordered from physical/link up through application. Today builds fluency in reading that structure, then applies it against four protocols you'll see in every single capture you ever take: DNS, TCP, HTTP, and TLS.

▸ why this mental model matters during analysis

When you "expand" a layer in the Packet Details Pane, you're literally peeling off one encapsulation wrapper to see what's inside — exactly the reverse of what the sending host's network stack did on the way out. A security analyst strips these layers away to isolate anomalies, pinpoint communication failures, or extract signs of compromise; knowing which header lives at which layer tells you exactly where in the tree to look for a given clue (MAC spoofing → L2, TTL anomalies → L3, port scanning → L4, credential theft → L7).

02

Capture filters vs. display filters — a distinction the basics often skip

Two different filter languages, two different moments, easy to confuse as a beginner.

Capture filters (BPF syntax)

Set before capture starts, in the capture options dialog. Uses Berkeley Packet Filter syntax — different from display filter syntax entirely (e.g. host 192.168.1.50, port 80). Packets that don't match are never written to the capture at all — gone permanently, not just hidden.

Display filters (the focus of today)

Applied after packets are already captured, in the filter bar at the top of the Wireshark window. Uses Wireshark's own filter syntax (e.g. ip.addr == 192.168.1.50, tcp.port == 80). Non-matching packets are hidden from view, not deleted — you can always clear the filter and see everything again.

▸ the beginner trap this causes

Typing ip.addr == 192.168.1.50 into the capture filter field will produce a syntax error or silently fail, because that's display-filter syntax, not BPF. Today's entire filter library below assumes you're already capturing (or working from a saved .pcap) and applying these as display filters.

03

Logical operators

The connective tissue for every filter below.

  • and / && — both conditions must be true
  • or / || — either condition may be true
  • not / ! — excludes matching traffic
ip.addr == 10.0.0.5 && tcp.port == 80 // both must match
http || dns // either protocol
!arp && !ssdp // hide common noise protocols
04

Isolating host & port traffic

The first thing you reach for once an IP or port is suspect.

⌗ All traffic to/from a suspicious IP

ip.addr == 192.168.1.50

⌗ Directional traffic — checking for exfiltration

ip.src == 192.168.1.50 and ip.dst == 203.0.113.5

Direction matters: ip.addr matches the host as either sender or receiver, while ip.src/ip.dst pin down which side it's on — essential when you specifically care about outbound flow versus inbound.

⌗ Filtering by Layer 4 ports

tcp.port == 80 or tcp.port == 443
tcp.dstport == 22 // destination only — reconnaissance often targets this specifically

tcp.port matches either source or destination port; tcp.srcport/tcp.dstport narrow to one side — the same source/destination distinction as IP addresses above, just one layer up the stack.

05

Content & string matching

Cleartext protocols hand you the indicator directly, if you know to look.

frame contains "password"

Searches the raw bytes of the entire frame for the literal string — works regardless of which layer or protocol field the match falls in, which makes it a good first-pass net but a noisy one (it'll also match the string inside unrelated binary data that happens to contain those bytes).

matches "select.*from"

Wireshark's matches operator applies a regular expression — useful for catching variations (case, spacing, surrounding context) that an exact contains match would miss. select.*from here is a crude SQL-injection-attempt detector, catching SQL keywords appearing together in a request body or URL.

▸ contains vs. matches, when to reach for which

Use contains for a known, fixed literal string — it's faster and simpler. Reach for matches only when you genuinely need pattern flexibility (variable casing, optional characters, multiple keyword variants) — regex evaluation is more expensive on large captures, so it's a precision tool, not a default.

06

Protocol deep dive — DNS

Often the very first signal of compromise, because malware needs to resolve a C2 domain before it can call home.

What to search: start with dns, then narrow:

dns.flags.response == 0 // queries only
dns.flags.response == 1 // answers only
dns.qry.type == 16 // TXT records — often abused for data exfiltration

① How to read the fields

  1. Apply dns and select a packet. Expand the Domain Name System layer in the Packet Details Pane.
  2. Check the Transaction ID A query and its matching response share the same Transaction ID. A response with no matching prior query — or two responses for one query — is a classic DNS spoofing/cache-poisoning signal.
  3. Expand Queries — inspect dns.qry.name Attackers encode payloads into subdomain labels to slip past firewalls that only inspect domain allow-lists at the registered-domain level, e.g. afe394b92c.attacker.com. An anomalously long, high-entropy-looking subdomain string is the signature to watch for.
  4. Expand Answers — check TTL Domains tied to fast-flux infrastructure rotate IPs rapidly and advertise ultra-low TTLs (often under 60 seconds) so DNS caches expire almost immediately, keeping the actual hosting IP a moving target.
▸ why DNS tunneling subdomains actually look the way they do

DNS labels can only contain a limited character set and have a 63-character length limit per label. To smuggle arbitrary binary data (a stolen file, a C2 command) through DNS, tunneling tools encode it — typically Base32 or Base64 — into that limited character set, then chop the result into label-sized chunks. That encoding is exactly why the resulting subdomain looks like meaningless alphanumeric noise rather than a real word: it's not random, it's binary data wearing a DNS-label costume.

07

Protocol deep dive — TCP & the three-way handshake

The flag pattern underneath every reliable connection, and the same pattern a port scan abuses.

tcp.flags.syn == 1 and tcp.flags.ack == 0 // new connection requests only
tcp.flags.reset == 1 // aborted connections
tcp.analysis.retransmission // latency or packet loss indicator

① The three-way handshake, packet by packet

Client
SYN seq=0
Server
Client
SYN, ACK seq=0, ack=1
Server
Client
ACK seq=1, ack=1
Server
  • Packet 1 [SYN] — client. Look in Flags for Syn: Set. Sequence number 0 (relative numbering, Wireshark's default display).
  • Packet 2 [SYN, ACK] — server. Both Syn: Set and Acknowledgment: Set.
  • Packet 3 [ACK] — client. Acknowledgment: Set. Connection now established.
▸ why the sequence/ack numbers actually matter, not just the flags

Sequence and acknowledgment numbers are how TCP guarantees reliable, ordered delivery — each side tracks exactly which bytes the other has confirmed receiving, so missing or out-of-order data can be detected and retransmitted. When Wireshark flags tcp.analysis.retransmission, it's specifically because a sequence number it already saw is appearing again — the sender never got the expected ACK in time and is resending, which is exactly the mechanic that lets TCP recover from packet loss without the application layer ever knowing it happened.

② Anatomy of a port scan in the capture

Apply tcp.flags.syn == 1. A single external IP sending many SYN packets to sequential destination ports within seconds, each immediately answered by a RST from your host, is an active reconnaissance scan in progress — this is literally what a SYN scan (Nmap's -sS, covered on Day 11) looks like from the receiving end's packet capture.

③ Following streams

Right-click any TCP packet → Follow → TCP Stream. This stitches every fragmented packet belonging to that conversation into one chronological, human-readable transcript — the fastest way to actually read what was said in a plaintext exchange, rather than piecing it together packet by packet.

08

Protocol deep dive — HTTP

Cleartext by default — an ideal hunting ground precisely because nothing is hidden.

http.request.method == "POST" // data transmissions: logins, uploads
http.response.code == 404 // scanning/enumeration indicator
http.user_agent matches "sqlmap" // automated attack tool signature
  1. Filter by http Check the request method (GET/POST) and response code (200 OK, 302 redirect, etc.) directly in the packet list.
  2. Expand the HTTP layer Inspect the Host and User-Agent fields. Real browsers have long, specific User-Agent strings; web shells and automated download scripts often use bare tool signatures like Wget/version or curl/version — or sometimes none at all.
  3. Inspect the full request URI Check for directory traversal sequences (../../etc/passwd) or SQL injection payloads embedded directly in URL query parameters.
▸ a generic User-Agent isn't proof of malice on its own

Plenty of legitimate automation — health checks, internal monitoring, API integrations — also uses bare curl/Wget user agents. Treat it as a flag worth correlating with destination, request pattern, and timing, not a standalone verdict.

09

Protocol deep dive — TLS / HTTPS

You can't read the encrypted payload without the key — but the handshake itself leaks a surprising amount.

tls.handshake.type == 1 // Client Hello
tls.handshake.type == 2 // Server Hello
tls.handshake.extensions_server_name // extract the SNI domain
  1. Filter by tls or ssl Locate the Client Hello packet first — it's the opening move of every TLS session.
  2. Expand TLS → Handshake Protocol → Extensions → server_name (SNI) This reveals the actual domain the client is connecting to, sent before encryption is established for the rest of the session. Compare it against DNS records from Phase 06 to spot mismatches.
  3. Locate the Server Hello and expand the certificate Check the Issuer and Validity Period. Self-signed certificates, certs from unfamiliar authorities, or validity windows lasting only days are characteristic of short-lived, disposable C2 infrastructure.
▸ why SNI is visible at all, if TLS is "encrypted"

This trips people up constantly: TLS encrypts the session content, but the Client Hello — the very first message, before any shared encryption key exists yet — is sent in the clear by necessity. The server needs to know which domain's certificate to present (a single IP can host many HTTPS domains), and the client has no encryption key to use yet at that point in the handshake. SNI is part of that initial cleartext exchange, which is exactly why DPI-capable firewalls (Day 10's NGFW discussion) can fingerprint destination domains without ever decrypting the actual traffic that follows.

10

SOC analyst playbook

When an alert fires, a PCAP is ground truth. This is the sequence — start broad, then narrow.

  1. Check high-level statistics first Don't write a single filter yet. Statistics → Conversations, sort the TCP or IPv4 tab by bytes transferred. A single host suddenly uploading gigabytes to an unfamiliar external IP is your first exfiltration signal, found before you've inspected a single packet field.
  2. Profile protocol distribution Statistics → Protocol Hierarchy breaks down what percentage of total traffic each protocol accounts for. An unprivileged segment (guest Wi-Fi, a workstation VLAN) showing a heavy chunk of SSH, RDP, or SMB traffic is a lateral-movement red flag worth investigating immediately.
  3. Extract artifacts and payloads Found a file download over unencrypted HTTP? File → Export Objects → HTTP... reconstructs the actual transferred binary directly from the packet stream — save it to an isolated sandbox for static analysis and hashing, never run it directly on an analysis workstation.
▸ pro tip for threat hunters

Always correlate Wireshark findings with host-based telemetry (Sysmon, EDR event logs). A suspicious HTTP POST to an unfamiliar IP becomes dramatically more significant the moment host logs show that connection was spawned directly from a malicious Office macro process — the network capture and the host log are two halves of the same story, and neither alone gives you the full picture.

Troubleshooting

What actually goes wrong during this lab.

Filter bar shows red/pink — "invalid filter" — even though the syntax looks right

Almost always a mismatch between capture-filter and display-filter syntax (Phase 02) — e.g. typing host 192.168.1.50 into the display filter bar, which expects ip.addr == 192.168.1.50 instead. Confirm which filter field you're actually typing into.

No TLS/HTTPS traffic decodable at all, every connection just shows as encrypted noise

That's expected and correct — without the session keys, Wireshark cannot decrypt TLS payload content, by design. You can still extract the SNI (Phase 09) and handshake metadata without decryption. Full decryption requires either the server's private key (rare to have) or a logged session-keys file from the browser (e.g. via the SSLKEYLOGFILE environment variable) — out of scope for today, but worth knowing exists for later.

Capturing on a Wi-Fi interface shows almost nothing, even with heavy network use

Most Wi-Fi adapters only show traffic addressed to/from your own machine by default, not the full broadcast domain like a wired switch span port would. This is normal for a laptop capture — you'll see your own traffic clearly, which is sufficient for today's lab; capturing other hosts' traffic on Wi-Fi requires monitor mode and is a separate, more advanced topic.

"Follow TCP Stream" shows garbled or empty content

If the underlying traffic is TLS-encrypted, the reconstructed stream will show encrypted bytes, not readable text — this is expected for HTTPS. Follow-stream readability only applies to plaintext protocols like HTTP, FTP, or Telnet.
