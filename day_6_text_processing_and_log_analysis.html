Day 06: Foundational Mastery

Linux Basics III: Text Processing & Log Analysis Workflow
Industry Insight 💡
A raw log file contains millions of events. As a Blue Teamer, your job isn't to read logs; it's to interrogate them. We use pipelines to chain tiny, single-purpose tools together to transform massive files into actionable threat intelligence.

1. The Anatomy of the Pipeline (|)

In Linux, the pipe operator | takes the Standard Output (stdout) of the command on the left and feeds it directly into the Standard Input (stdin) of the command on the right. This allows you to build a continuous assembly line for data.

[Raw Log] ➡️ grep ➡️ [Filtered Rows] ➡️ awk ➡️ [Specific Columns] ➡️ sort/uniq ➡️ [Actionable Intelligence]

2. Isolating the Threat with grep

grep (Global Regular Expression Print) searches line-by-line for a specific pattern. When analyzing an authentication log (/var/log/auth.log), you don't care about successful logins; you want to isolate failures.

# Search for lines containing the exact string "Failed password"
omnisec@soc:~$ grep "Failed password" /var/log/auth.log

# Output Example:
Oct 14 09:12:34 server sshd[1234]: Failed password for root from 192.168.1.50 port 44322 ssh2

Crucial Flags: -i (ignore case), -v (invert match - show lines that don't contain the word), -r (search recursively through folders).

3. Surgical Extraction with awk

While grep pulls out the whole row, awk pulls out specific columns. Logs are space-separated by default. Looking at our output above, the IP address (192.168.1.50) is the 11th word in the sentence.

# Pipe the grep output into awk, telling awk to print only the 11th field ($11)
omnisec@soc:~$ grep "Failed password" /var/log/auth.log | awk '{print $11}'

# Output:
192.168.1.50
192.168.1.50
10.0.0.5

4. Threat Pattern Recognition (sort & uniq)

Now we have a list of IPs, but an attacker brute-forcing a server will leave thousands of identical IPs. We need to count them to identify the worst offenders.

Rule of Linux: uniq can only remove duplicates if they are directly next to each other. Therefore, you must sort the data before passing it to uniq.

# 1. sort: alphabetically group IPs together
# 2. uniq -c: count adjacent duplicates
# 3. sort -nr: sort numerically (-n) in reverse/descending order (-r) to put the biggest attacker on top
omnisec@soc:~$ grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -nr

# Output (The Holy Grail of Log Analysis):
1542 192.168.1.50 <-- Massive Brute Force Attack
3 10.0.0.5 <-- Probably a user who forgot their password

5. Automating the Hunt (Bash Scripting)

SOC Analysts don't type these long pipelines manually every day. We wrap them in reusable shell scripts.

#!/bin/bash
# File: find_attackers.sh
# Purpose: Extracts top 5 attacking IPs from a provided log file

LOG_FILE=$1

echo "[*] Analyzing $LOG_FILE for SSH brute-force attempts..."
echo "--------------------------------------------------------"
grep "Failed password" "$LOG_FILE" | awk '{print $11}' | sort | uniq -c | sort -nr | head -n 5
echo "--------------------------------------------------------"
echo "[*] Analysis Complete. Consider blocking the top IPs."

Usage: chmod +x find_attackers.sh followed by ./find_attackers.sh /var/log/auth.log
