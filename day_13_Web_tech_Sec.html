Overview & why this comes first

Tools without framing produce noise. Framing without tools produces nothing. Day 13 is the framing.

It's tempting to skip straight to "fun" tools — Nmap, Burp, Metasploit — but every one of those tools answers a question, and you can't ask a good question without knowing what kind of assessment you're doing and what you're actually looking at. A vulnerability scan and a penetration test point the same tools at the same target and produce completely different deliverables, because the goal differs. Today fixes that framing problem before Day 14 gets you reading raw HTTP traffic and Day 15 gets you intercepting it with Burp Suite.

▸ the one-sentence version

Day 13 answers "what am I trying to find out, and where would it actually live in this application?" — everything from Day 14 onward answers "how do I go get it?"

01

The four assessment types

Same target, four different jobs. Knowing which one you're hired (or assigned) to do changes everything about your approach.

Assessment TypePrimary scope & goalTypical output
Vulnerability Assessment Identify and list known security flaws — automated, surface-layer, breadth over depth. A scored list of findings (e.g. a GVM/OpenVAS report) — see Day 11.
Penetration Testing Actively exploit vulnerabilities to verify real-world risk and depth of access — manual + automated, depth over breadth. A narrative report showing what was actually achievable, with proof of exploitation.
Security Auditing Check adherence to a specific regulatory framework or internal control set (PCI-DSS, ISO 27001, SOC 2, internal policy). A compliance checklist — pass/fail against named controls, not a vulnerability list.
Risk Assessment Analyze systemic threats against business assets and calculate overall business impact — likelihood × impact, not just technical severity. A risk register, often feeding into business/budget decisions rather than a patch list.
▸ the mistake almost everyone makes early on

Calling a vulnerability scan a "penetration test." A scanner like OpenVAS tells you a service might be vulnerable based on version-matching against known CVEs — it has not actually tried to break in. A real pentest takes that list and attempts exploitation to confirm the vulnerability is genuinely reachable and exploitable in this specific environment, not just theoretically present. Conflating the two is one of the fastest ways to lose credibility with a client or a hiring manager who actually knows the difference.

02

How they actually relate to each other

These aren't four competing options — they're frequently stages of the same larger engagement.

In practice, a full security program often runs all four in a loose sequence, at different cadences:

  1. Risk Assessment sets the priorities Which assets matter most to the business, and what's the acceptable risk tolerance? This usually happens at a program level, annually or per major project — not per-target.
  2. Vulnerability Assessments run continuously or on a schedule Cheap, automatable, broad coverage (this is exactly what your Day 11 OpenVAS lab produces) — the ongoing "what's currently exposed" signal.
  3. Penetration Testing happens periodically, targeted Take the highest-value findings from vulnerability assessments (or a fresh black-box engagement) and verify real-world exploitability — typically annual, or after major changes.
  4. Security Auditing runs against whatever regulatory clock applies Independent of the above — a compliance audit checks documented controls and processes exist and are followed, which can pass even on a system riddled with unpatched CVEs if those CVEs aren't covered by the specific control framework being audited (and vice versa).
▸ why this matters for your career path

SOC Analysts and Vulnerability Management roles live mostly in step 2. Penetration Testers and Red Teamers live in step 3. GRC (Governance, Risk & Compliance) roles live in steps 1 and 4. Knowing this map early helps you read job postings accurately — "pentester" and "vulnerability analyst" are not interchangeable titles even though outsiders often conflate them.

03

Frontend vs backend attack surface

Not just "what runs where" — what that location implies about who can tamper with it.

Frontend — client-side

HTML, CSS, and JavaScript that ships to and runs inside the user's own browser — fully visible and editable via DevTools. Anything enforced only in frontend code is not actually enforced — a user (or attacker) controls their own browser completely and can disable, rewrite, or bypass any client-side check.

  • Form validation that only checks in JS — trivially bypassed by sending the request directly
  • "Hidden" form fields and disabled buttons — visible and editable in the DOM
  • Client-side route guards (e.g. a React router redirect) — irrelevant if the underlying API has no server-side check

Backend — server-side

Server-side processing logic (Node.js, Python, PHP, Java, etc.) handling databases (SQL/NoSQL), business logic, and authentication — hidden behind the HTTP boundary. This is where real enforcement has to live, because the client cannot tamper with code it never receives.

  • Authentication and authorization checks
  • Input validation and sanitization before it ever touches a database query
  • Business rules (pricing, permissions, rate limits)
▸ the single most important takeaway from this section

"Security through obscurity" on the frontend is not security. If a check, a price calculation, or an access decision happens only in JavaScript that ships to the browser, treat it as already broken — the only question is how the attacker reaches the unprotected backend endpoint directly. This is the conceptual root of an enormous fraction of real-world web vulnerabilities, including most broken access control and IDOR (Insecure Direct Object Reference) findings you'll study later.

04

Why headers matter

HTTP response headers are a free intelligence leak in one direction, and a free defense layer in the other — and most apps get both halves wrong.

Response headers reveal server software, technology stacks, and — critically — whether the defensive headers that should be there actually are. Here's the working set worth knowing by name:

X-Powered-By Directly announces the backend framework (e.g. Express, PHP/8.1) — pure reconnaissance gift to an attacker, no defensive value. Best practice: disable it entirely.
Server Announces the web server software and often its exact version (e.g. Apache/2.4.52) — lets an attacker go straight to a CVE database for that exact build.
Content-Security-Policy Restricts what sources scripts, styles, and frames can load from — the single strongest defense against XSS payload execution even if injection occurs.
Strict-Transport-Security Forces the browser to only ever connect over HTTPS for this domain going forward — closes the window for downgrade/strip attacks on subsequent visits.
X-Content-Type-Options nosniff — stops the browser from guessing content types in a way that can turn an uploaded "image" into executable script in some legacy scenarios.
X-Frame-Options Controls whether the page can be embedded in an <iframe> elsewhere — its absence is a prerequisite for clickjacking attacks.
▸ reading headers is a two-sided exercise

You're checking for two different things at once: headers that are present and shouldn't be (information disclosure — X-Powered-By, Server with version strings) and headers that are absent and should be present (missing defenses — CSP, HSTS). A clean-looking response with no banners can still be missing every defensive header; both checks matter independently.

05

Where the OWASP Top 10 fits

The source doc didn't mention it, but you'll see it referenced constantly from here on — worth anchoring now.

The OWASP Top 10 is a regularly-updated, community-curated list of the most critical web application security risks, maintained by the Open Worldwide Application Security Project. It's not a checklist of specific bugs — it's a list of categories of risk (e.g. "Broken Access Control," "Injection," "Security Misconfiguration") that almost every assessment type from Phase 01 ends up organizing its findings around, because clients, auditors, and other security teams all share this common vocabulary.

  • A vulnerability assessment finding often maps directly to an OWASP category (e.g. a missing CSP header → "Security Misconfiguration").
  • A penetration test report typically structures its executive summary around which OWASP categories were confirmed exploitable.
  • A security audit against a framework like PCI-DSS will frequently cross-reference OWASP categories as evidence of secure coding practice.

06

Passive vs active recon

A distinction the source material uses implicitly (Task 1 vs Task 2) but never names — naming it now will save confusion in every later recon-heavy day.

Passive recon

Gathering information without sending traffic the target would notice as unusual — or sometimes without touching the target at all. Browsing cached pages, reading public DNS records, checking certificate transparency logs, using third-party fingerprinting services like Wappalyzer/BuiltWith that analyze a page you load normally.

Active recon

Directly interacting with the target in a way that generates logs, touches the server, and could theoretically be detected — a curl request, a port scan, a directory brute-force. Still non-destructive at this stage, but no longer invisible.

Task 1 below (curl -I) is technically active — it's a real request hitting the real server. Task 2 (Wappalyzer/BuiltWith) is passive — you're just analyzing a page you'd load anyway. In a real engagement with defined scope and rules of engagement, this distinction determines what you're allowed to do before formal authorization kicks in versus what requires it.

▸ scope reminder

Every command in this guide should be run against example.com as written (it's a real, intentionally-public IANA test domain meant for exactly this), or against infrastructure you own or have explicit written authorization to test. Don't repoint these at targets you don't control.

07

Task 1 — active header enumeration via curl

A single quiet request, read carefully.

bash
$ curl -I https://example.com

HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 1256
Date: Sat, 27 Jun 2026 10:02:11 GMT
Cache-Control: max-age=604800
Server: ECAcc (dcd/7D5A)
X-Cache: HIT
Vary: Accept-Encoding

Walking through what to actually check, line by line:

  1. Status line Confirms the server responded and how — a 200 here, but watch for 403/redirect chains on real targets, which can themselves reveal WAF or auth-gate presence.
  2. Server / X-Powered-By presence example.com's response above is deliberately sparse — most production targets will leak more. Note exactly what's disclosed: software name, version number, OS hints.
  3. Missing security headers Notice there's no Content-Security-Policy, Strict-Transport-Security, or X-Frame-Options in this response — exactly the "absent and should be present" check from Phase 04.
  4. Caching headers Cache-Control and X-Cache reveal CDN/proxy presence — useful context for understanding the actual infrastructure in front of the origin server.
▸ leveling this up

Add -v instead of -I to also see the TLS handshake details (negotiated protocol version, cipher suite) — relevant once you start checking targets for deprecated TLS 1.0/1.1 support, a finding that shows up constantly in real vulnerability assessments.

The point of this task, tied back to Phase 01: a verbose server banner with an outdated version string is exactly the kind of finding that feeds a Day-11-style vulnerability scan or a Day-12 exploit-database lookup — header enumeration is the manual, single-target version of what an automated scanner does across an entire range.

08

Task 2 — passive footprinting via browser extension

Let someone else's fingerprinting database do the version-matching for you.

Wappalyzer

Browser extension that analyzes a loaded page's HTML, JS globals, cookies, and headers against a large signature database to identify CMS platforms, JS frameworks, analytics tools, and server technologies — all without sending a single extra request to the target.

BuiltWith

Similar fingerprinting, with a stronger emphasis on infrastructure-level detail: CDN providers, hosting providers, embedded third-party APIs, and advertising/analytics integrations — useful for mapping an organization's broader technology footprint, not just the single page in front of you.

  1. Install the Wappalyzer extension in your browser and navigate to a target you're authorized to assess.
  2. Click the extension icon — it surfaces a categorized list (CMS, JS framework, web server, analytics, etc.) instantly, with no action on your part beyond loading the page.
  3. Cross-reference against Task 1's curl output — does the Server header agree with what Wappalyzer detected? Discrepancies are worth investigating; they sometimes indicate a reverse proxy or CDN masking the true origin stack.
  4. Repeat with BuiltWith for the infrastructure-level view — CDN, hosting provider, DNS provider — building a fuller picture of the target's technology supply chain.
▸ why "free" tools like this matter professionally

Passive fingerprinting tools are exactly what a real-world attacker (or a bug bounty hunter doing initial recon under a tight scope) reaches for first — they're zero-risk, zero-noise, and often reveal outdated or unusual technology choices that narrow your next steps dramatically before you send a single active probe.

DevTools cheat sheet — properly explained

Not just what each tab is called — what specifically to look for in each one.

Elements tab

Shows the live, rendered DOM — which can differ from the original HTML source after JavaScript has run. Look for:

  • Hidden input fields (type="hidden") carrying IDs, tokens, or flags the UI doesn't surface visually
  • Disabled buttons or inputs — check if disabling is purely a CSS/attribute toggle you can remove client-side
  • Inline event handlers and embedded <script> blocks revealing internal logic or endpoint names
  • HTML comments occasionally left in by developers containing internal notes or stale debug info

Network tab

The full record of every request the page makes. Look for:

  • XHR/fetch calls to API endpoints not visible anywhere in the rendered page — often the real attack surface for SPAs
  • JSON request/response bodies revealing the actual data shape and field names the backend expects
  • Status codes across requests — a stray 403/500 reveals an endpoint behaving unexpectedly under normal use
  • Request headers showing auth tokens, API keys, or session identifiers in transit

Console tab

Client-side JavaScript execution logs. Look for:

  • Unhandled errors and stack traces — these frequently leak internal file paths, framework versions, or variable names
  • Developer console.log() debug statements accidentally left in production
  • The ability to directly execute arbitrary JS in the page's own context — useful later for manually testing DOM-based XSS sinks

Application / Storage tab

Not in the original list, but essential alongside the other three — shows cookies, localStorage, and sessionStorage. Look for:

  • Session tokens stored in localStorage (vulnerable to theft via any XSS, unlike HttpOnly cookies — see Day 14)
  • Cookie flags — is Secure or HttpOnly actually set?
▸ the habit that makes this stick

Don't just read this list — open DevTools on three sites you use daily right now and find one concrete example of each bullet point above. Recognition speed on real, messy production sites is the actual skill; this cheat sheet is just the index.

11

Beginner tips

What trips people up specifically on Day 13.

The four assessment types feel like trivia, not a skill

They click once you read a few real reports. Search for a public sample penetration test report and a public sample vulnerability assessment report side by side (many security firms publish redacted samples) — the structural difference in how findings are framed becomes obvious fast, far more than memorizing the table above.

Wappalyzer shows nothing, or shows wildly outdated info

Signature-based fingerprinting can lag behind a site's actual current stack, and some frameworks are deliberately obfuscated against this kind of detection. Treat any single tool's output as a hypothesis to confirm against a second source (the curl headers, page source view, response timing patterns) — not ground truth on its own.

Not sure which DevTools finding is actually "interesting" vs. just normal

At this stage, "interesting" just means "I didn't expect this to be visible." A hidden field with a real database ID, an API call with no visible UI trigger, a console error mentioning an internal hostname — none of these are exploits yet, they're just things worth writing down and coming back to once you have the tools (Burp Suite, Day 15 onward) to actually act on them.
