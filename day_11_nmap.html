Overview & objective

Nmap — Network Mapper — the most widely used network discovery and security auditing tool in the field.

Today's objective is hands-on fluency with Nmap for the core reconnaissance loop: find live hosts, find open ports on those hosts, identify what's actually running on those ports, and pull whatever version/OS fingerprinting detail is available. Nmap is essential for:

  • Network discovery
  • Port scanning
  • Service enumeration
  • Operating system detection
  • Vulnerability assessment (via NSE)

01

Why Nmap matters

You cannot secure — or attack — what you don't know exists.

Before securing a system, you must understand what is actually exposed. Nmap helps security professionals discover live hosts, identify open ports, determine running services, detect service versions, identify operating systems, and surface potential vulnerabilities — all from the network side, without needing any credentials or prior access.

▸ where this sits in a real engagement

In real-world engagements, Nmap is almost always the first tool used during reconnaissance — the phase where you build a map of the target before deciding what to attack, what to scan more deeply (a GVM/OpenVAS-style vulnerability scan), or what to report. Everything downstream depends on this map being accurate.

02

What is port scanning?

The mechanical question underneath everything else today: is anything listening here?

Port scanning is the process of checking which ports on a target system are open, closed, or filtered. A "port" here is just a numbered endpoint (0–65535) that a service binds to — port 22 for SSH, port 80 for HTTP, port 3306 for MySQL, and so on by convention, though nothing stops a service from running on an unconventional port. Understanding open ports helps security professionals identify running services, potential attack surfaces, misconfigurations, and security risks.

▸ the mental model

Think of a target machine as a building with thousands of numbered doors. Most are locked (closed) or don't exist in any meaningful sense for this building (filtered by a firewall before you even reach the door). A handful are unlocked because something inside is actively expecting visitors (open, with a service listening). Port scanning is systematically knocking on every door to map which ones respond.

Common Nmap scan types — explained mechanically

Not just flags to memorize — what's actually happening on the wire for each one, and why that changes stealth.

-sTTCP Connect ScanFull TCP handshakeNo root privileges availableLow
-sVVersion DetectionDetect service versionsService enumerationMedium
-OOS DetectionDetect operating systemHost profilingMedium
-AAggressive ScanOS, version, scripts, tracerouteDetailed reconnaissanceLow
-pPort SelectionScan specific portsTargeted scanningN/A
--scriptNSE ScriptsRun scriptsAdvanced enumerationVaries

⚙ Why SYN scan (-sS) is stealthier than Connect scan (-sT)

This is the single most-asked "why" in this table, and it's worth understanding the actual TCP mechanics rather than memorizing "SYN = stealthy":

-sT — full connect scan

Completes the entire three-way TCP handshake: SYN → SYN/ACK → ACK. Because a full connection is established, the operating system's normal connection logging captures it — every connect scan shows up in application-level logs on the target as a real connection attempt. This is also why it doesn't require root/admin privileges: it's just asking the OS's normal socket API to connect like any other program would.

-sS — SYN ("half-open") scan

Sends only the initial SYN packet. If the target replies SYN/ACK, Nmap knows the port is open — and immediately sends a RST instead of completing the handshake with an ACK. Because the connection is never actually established, many applications never log it at the application layer (though it can still appear in firewall/IDS logs watching raw packets) — hence "stealthier." It requires raw packet crafting, which is why it needs root privileges.

▸ "stealth" is relative, not absolute

A SYN scan is stealthier relative to a full connect scan against basic logging — it is not invisible. Any modern IDS/IPS or even a default firewall logging dropped/reset connections will still see a SYN scan's traffic pattern (many ports, rapid SYNs, no completed handshakes) and flag it as reconnaissance. Don't treat "-sS" as a magic invisibility flag.

⚙ Why -A is "Low" stealth despite being the most useful single flag

-A is shorthand for enabling OS detection, version detection, default NSE scripts, and traceroute all at once. Each of those individually adds detectable traffic patterns (OS detection alone sends a distinctive battery of unusual packets to provoke fingerprint- revealing responses) — stacking all four makes an aggressive scan extremely easy to fingerprint as reconnaissance, even though it's enormously useful for fast, thorough lab learning where stealth doesn't matter yet.

04

Key options & flags

The handful of modifiers you'll reach for in nearly every real scan.

$ Skip host discovery

bash
$ nmap -Pn <target>

Purpose: Assume the target is online and skip the initial ping/ARP-based alive-check. Useful when ICMP is blocked by a firewall — without -Pn, Nmap may conclude a perfectly live, listening host is "down" simply because it doesn't respond to ping, and skip scanning it entirely.

$ Faster scanning

bash
$ nmap -T4 <target>

Purpose: Increase scanning speed by reducing inter-probe delays and timeouts — covered fully in the timing section below. Useful in lab environments with no IDS to worry about triggering.

$ Save results

bash
$ nmap -oN scan_results.txt <target>

Purpose: Save output to a file for documentation and reporting. -oN writes normal human-readable format; swap to -oX for structured XML (parseable by other tools and report generators) or -oA to write normal, XML, and grepable formats all at once with one filename stem.

$ Scanning a range instead of a single host

bash
# CIDR notation — scan an entire /24 subnet
$ nmap 192.168.56.0/24

# explicit range
$ nmap 192.168.56.10-50

# multiple specific targets
$ nmap 192.168.56.101 192.168.56.102 192.168.56.105

Worth knowing even though today's lab targets a single Metasploitable2 VM — real engagements almost always start with a range, not a single known IP.

05

Timing templates (-T0 through -T5)

The source material mentions -T4 once with no context — here's the full scale, since you'll see all six in the wild.

-T0ParanoidOne probe at a time, very long delays — designed to evade IDS detection over hours/days. Rarely used outside red team OPSEC-critical work.
-T1SneakySimilar intent to T0, somewhat faster — still built for evasion over a long time window.
-T2PoliteSlows down to reduce bandwidth/load impact on the target — considerate of fragile or low-bandwidth targets, not primarily about evasion.
-T3NormalThe default if no -T flag is given — a reasonable general-purpose balance.
-T4AggressiveAssumes a fast, reliable network — shorter timeouts, more parallelism. The standard choice for lab work and CTFs where speed matters more than subtlety.
-T5InsaneMaximum speed, sacrifices accuracy — packets sent so fast that slower or heavily-loaded targets may give incomplete/incorrect results. Use only when speed matters more than reliability.
▸ the practical rule

In your home lab against Metasploitable2: -T4 is the right default — fast enough to keep labs from dragging, reliable enough not to miss results. Save T0–T2 for when you're specifically practicing evasion technique against a target with active monitoring, which comes later in the roadmap.

06

NSE scripts under the hood

The source material uses --script vuln in the lab without ever explaining what NSE actually is — here's the missing piece.

NSE (Nmap Scripting Engine) is a Lua-based scripting framework built into Nmap that lets it go far beyond "is this port open" into actual service interrogation — banner grabbing, brute-forcing weak credentials, checking for specific known vulnerabilities, and more. Scripts are organized into categories, and you'll use a handful constantly:

--script vuln

Runs every script tagged in the "vuln" category — checks for a wide range of known vulnerabilities against detected services. This is what the lab's Step 6 uses; it's effectively a lightweight, Nmap-native cousin of what a dedicated scanner like OpenVAS does at much greater depth.

--script default (-sC)

A curated, safe-by-default set of scripts covering common, low-risk information gathering — this is what's bundled automatically inside -A.

--script discovery

Scripts focused on gathering more information about the network and host without targeting specific vulnerabilities — e.g. SMB share enumeration, DNS zone info.

--script <specific-script-name>

Run one named script directly — e.g. --script smb-vuln-ms17-010 to check specifically for EternalBlue exposure, once you know exactly what you're looking for.

▸ vuln-category scripts are not always safe

Some scripts in the vuln category are intrusive — a handful can actually crash a fragile or unpatched service while testing for the vulnerability (this overlaps with the same caution OpenVAS's "Full and fast ultimate" profile requires). Fine against Metasploitable2, which is built to be hammered on; think twice before running --script vuln against anything more fragile or production-adjacent.

Reading port states

Three words that mean three completely different security stories.

StateExampleWhat actually happened on the wireSecurity implication
Open 22/tcp open ssh Target replied SYN/ACK (or completed handshake) — a service is actively bound and listening. Real attack surface — this is a service you can actually try to interact with, authenticate to, or exploit.
Closed 443/tcp closed https Target replied with a RST — reachable, but explicitly nothing listening. Confirms the host is alive and the network path is open, but this specific port is not currently usable.
Filtered 80/tcp filtered http No response at all (or an ICMP unreachable) — a firewall or filtering device is dropping the probe before it reaches the actual service. You genuinely don't know if a service is there. Filtered ports often deserve a second look with a different scan technique or source port before writing them off.
▸ why "filtered" deserves more attention than it gets

Closed tells you something definite (no service there). Filtered tells you almost nothing — it could be a heavily-firewalled critical service or genuinely nothing at all. Beginners often treat filtered the same as closed and move on; experienced testers treat a filtered port as an open question worth one more probe with different timing, source port, or scan type before concluding anything.

08

Lab setup — Metasploitable2

An intentionally vulnerable Linux machine, purpose-built for exactly this kind of training.

  1. Download Metasploitable2 From a reputable mirror (Rapid7's official SourceForge listing is the canonical source).
  2. Extract the downloaded ZIP file You'll get a folder containing a .vmdk virtual disk file.
  3. Open VirtualBox and create a new virtual machine Type: Linux, version: Ubuntu (older, 32-bit) — Metasploitable2 is built on an old Ubuntu base, so don't worry about matching modern specs exactly.
  4. Use the extracted VMDK file as the hard disk When prompted for a virtual hard disk, choose "Use an existing virtual hard disk file" and point it at the .vmdk you extracted.
  5. Configure the network adapter Set it to Host-Only Adapter or Internal Network — both isolate the VM to a private virtual segment shared with your Kali VM, keeping it off your real LAN and the internet entirely. This is the single most important setup step for both safety and for your Kali box actually being able to reach it.
  6. Start the VM Boot can take a minute or two on the old kernel — this is normal, not a hang.
▸ default credentials

Username: msfadmin · Password: msfadmin — deliberately weak and well-documented, since the entire point of this VM is to be broken into.

▸ never bridge this VM's network adapter

Metasploitable2 is intentionally riddled with unpatched, severe vulnerabilities. Setting its network adapter to Bridged (instead of Host-Only/Internal) would expose every one of those vulnerabilities to your actual home or office network. Double-check this setting before powering on for the first time.

09

Hands-on scanning lab (2–3 hours)

Six steps, each building on the last — from finding the target to surfacing real vulnerabilities.

① Identify the target IP address

Inside Metasploitable2, run:

msfadmin@metasploitable
$ ip addr

2: eth0: ... inet 192.168.56.101/24 ...

Record the IP address — example used throughout: 192.168.56.101.

② Perform a basic scan

kali — bash
$ nmap 192.168.56.101

PORT     STATE SERVICE
21/tcp   open  ftp
22/tcp   open  ssh
23/tcp   open  telnet
25/tcp   open  smtp
80/tcp   open  http
139/tcp  open  netbios-ssn
445/tcp  open  microsoft-ds
3306/tcp open  mysql

Purpose: Discover open ports and identify basic services using Nmap's default top-1000-port scan with no special flags.

③ SYN scan with version detection

kali — bash
$ sudo nmap -sS -sV 192.168.56.101

21/tcp open  ftp     vsftpd 2.3.4
22/tcp open  ssh     OpenSSH 4.7p1 Debian 8ubuntu1
23/tcp open  telnet  Linux telnetd

Purpose: Perform stealthier half-open scanning while also pulling specific service version strings — vsftpd 2.3.4 here is a famous, deliberately-planted backdoored version on this VM, exactly the kind of detail you'd cross-reference against a CVE database next.

④ Aggressive scan

kali — bash
$ sudo nmap -A -T4 192.168.56.101

Purpose: Detect operating system, run default NSE scripts, perform a traceroute, and collect detailed information — the broadest single-command sweep available, at the cost of subtlety as covered in Phase 03.

⑤ Vulnerability script scan

kali — bash
$ sudo nmap -sV -p 21,22,80,443 --script vuln 192.168.56.101

Purpose: Identify known vulnerabilities, enumerate services, and practice security assessment technique — narrowing to specific ports with -p first keeps this from taking forever, since the vuln script category is large.

▸ document as you go, not after

Pipe every step's output into -oN files as you run them (Phase 04) — re-running scans later to recreate a screenshot you forgot to save is the single most common time-waster in this kind of lab exercise.

10

Safety & legal guidelines

Non-negotiable, every single time you open a terminal with Nmap in it.

▸ scope, stated plainly

Only scan: your own Kali VM, Metasploitable2, systems you personally own, or environments you have explicit written authorization to test.
Never scan: public IP addresses, company networks without permission, or any system whose ownership/authorization you're not certain of.
Unauthorized scanning may violate computer-misuse laws (e.g. the CFAA in the US, the Computer Misuse Act in the UK, and equivalent statutes elsewhere) or organizational policy, independent of whether anything was actually exploited.

Even "just" a port scan against a system you don't control can trigger an incident response process on the receiving end, regardless of intent — treat scope as a hard boundary, not a guideline.

Troubleshooting

The failure modes that actually happen during this specific lab.

Kali can't reach Metasploitable2 at all (100% packet loss)

Almost always a network adapter mismatch — both VMs need to be on the same Host-Only or Internal Network, not just "an" internal network. In VirtualBox, check Settings → Network on both VMs and confirm the adapter name matches exactly (e.g. both set to vboxnet0, or both set to the identical Internal Network name). A one-character mismatch silently puts them on two separate isolated segments.

nmap reports the host as "down" despite the VM clearly being on and running

Metasploitable2's firewall/network stack may not respond to the ICMP ping Nmap's host discovery relies on by default. Add -Pn to skip the alive-check and scan regardless — see Phase 04.

"sudo: command not found" or permission denied running -sS

SYN scan needs raw socket access, which requires root. If you're not running as root already on Kali (common on fresh installs that default to a non-root user), prefix with sudo as shown throughout the lab, or fall back to -sT (Connect scan) which works without elevated privileges.

Scan takes far longer than expected

Default Nmap behavior with no -p flag scans the top 1000 most common ports, not all 65535 — already reasonably fast. If a scan is dragging, check whether you accidentally added -p- (all ports) or are scanning a wide CIDR range without intending to; add -T4 and/or narrow the port range with -p for lab purposes.

Every port shows as filtered, nothing shows as open

Check Metasploitable2's own host-based firewall state and confirm the Host-Only/Internal Network adapter is actually active and not still defaulting to NAT (the VirtualBox default for new VMs, easy to forget to change). A consistent "filtered" wall across every port usually means something is intercepting traffic before it reaches the VM at all, rather than the VM's services being genuinely firewalled individually.
