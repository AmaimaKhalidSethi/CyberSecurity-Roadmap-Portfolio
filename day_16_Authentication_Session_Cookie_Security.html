Authentication, Sessions & Cookies Security

Most real-world breaches don't start with a clever exploit — they start with a stolen session token or a login system that trusted the wrong thing. Today builds the full mental model: who you are, what you're allowed to do, and exactly how a server keeps remembering both across a stateless protocol.

00

Overview

Today's objective: understand how websites identify users, manage logged-in sessions, and protect authentication data.

Authentication, authorization, sessions, and cookies sit underneath nearly every serious real-world web attack — far more breaches trace back to a weak login system or a poorly managed session than to an exotic zero-day. Getting this layer genuinely solid pays off across everything else in the roadmap.

01

Authentication — "who are you?"

Verifying identity, before anything else happens.

Authentication answers exactly one question: who are you? Common methods:

  • Username & password
  • PIN code
  • Fingerprint / face recognition
  • Multi-factor authentication (MFA — covered fully in Phase 08)
▸ example

Logging into Gmail with your registered email address and password is authentication, full stop — nothing about permissions has been decided yet.

02

Authorization — "what can you do?"

A separate question, answered only after identity is already confirmed.

After authentication succeeds, the application checks which resources or privileges this specific identity is permitted to access.

Administrator

Can delete users, manage platform settings, view system logs.

Normal user

Can only view their own profile, edit personal details, change their own password.

▸ crucial rule

Authentication always happens before authorization. A system can never correctly answer "what are you allowed to do" without first answering "who are you" — skipping or weakening the first question breaks every guarantee built on top of it.

03

Authentication vs. authorization, side by side

The distinction that gets conflated constantly in casual conversation — worth nailing down precisely.

AuthenticationAuthorization
Verifies identityDetermines permissions
"Who are you?""What can you do?"
Happens firstHappens after successful authentication
▸ why this distinction has real teeth in security testing

A huge category of real vulnerabilities — broken access control, IDOR (Insecure Direct Object Reference), privilege escalation — exist precisely in the gap between these two checks: the application correctly confirms who you are, but fails to properly verify what you're allowed to touch once you are. Authenticating successfully as a normal user and then finding you can still reach an admin-only endpoint is an authorization failure, not an authentication one — and that distinction determines what you'd actually report and how.

04

Sessions — the full lifecycle

HTTP has no memory of its own. A session is the workaround.

Because HTTP is stateless, without sessions a user would have to re-authenticate on every single page load. A session is how the server fakes that memory.

1
User authenticatesSubmits credentials via login form.
2
Server validates and creates a sessionCredentials check out; a server-side session record is created.
3
Server generates a unique Session IDA token referencing that server-side session record — see Phase 05 for why this generation step is the single most security-critical moment in the whole lifecycle.
4
Browser stores the Session ID in a cookieDelivered via a Set-Cookie response header.
5
Browser auto-includes it on every subsequent requestVia the Cookie request header — this is what makes the user "stay logged in" across pages.
05

Why session IDs must be unpredictable, not just unique

The source material says "unique" — the actual requirement is much stronger than that, and the gap matters.

example session cookie
Cookie: PHPSESSID=ad3f7e21c18d55a7b8f93e

A counter (1, 2, 3, 4...) would satisfy "unique" perfectly — every value is different from every other value. It would also be a catastrophic security failure, because an attacker who knows their own session ID was 10042 could simply try 10041, 10040, and so on, and land directly inside other users' active sessions. The actual requirement is cryptographic unpredictability — given any number of valid session IDs already observed, an attacker should have no practical way to guess another valid one. This is why session IDs are generated from a cryptographically secure random number generator with enough entropy (effectively, enough random bits) that guessing one by brute force is computationally infeasible.

▸ a session ID is exactly as valuable as a password

Anyone who obtains a valid session ID can impersonate that user completely, without ever knowing their actual password — no login form, no MFA prompt, nothing. This is precisely why session IDs deserve every bit of the protection (Secure, HttpOnly, SameSite — Phase 06) that a password would get, and why session hijacking is such a consistently effective attack class.

06

Cookies, types & security attributes

Small key-value records, with a handful of flags that determine almost everything about how safely they're handled.

Cookies are small key-value string records the server stores locally inside the user's browser. Common uses:

  • Login session tokens
  • User layout preferences
  • Persistent shopping carts
  • Language and localization choices
  • Tracking and visitor analytics

Session Cookie

Exists only while the browser stays open — cleared entirely once the browser process closes.

Persistent Cookie

Remains on disk until its specified expiration date — what powers "Remember Me" functionality.

Secure attribute

Commands the browser to transmit the cookie only over HTTPS, blocking cleartext exposure on any HTTP fallback.

HttpOnly attribute

Blocks the cookie from being read by client-side JavaScript — the direct mitigation against token theft via XSS.

▸ Secure and HttpOnly are flags on a cookie, not separate cookie types

A single cookie can be persistent and Secure and HttpOnly all at once — these are independent attributes layered onto whichever lifetime type (session or persistent) the cookie already has, not a third category of cookie.

07

SameSite, worked examples

The three values are easy to define and easy to misapply — here's what actually triggers each one.

Strict Never sent on any cross-site request, full stop — including a user clicking a link to your site from an external page. Worked example: a user on a forum clicks a link to your banking site; with Strict, even that legitimate top-level navigation arrives with no session cookie attached, and the user appears logged out until they navigate within the site again.
Lax Sent on cross-site top-level safe navigations (clicking a link, typing a URL — effectively GET requests that change the top-level page) but withheld on cross-site subrequests like images, iframes, or background fetch/XHR calls, and withheld on cross-site POST. Worked example: the same forum-link click now correctly carries the session cookie, since it's a top-level GET navigation — but a malicious page's hidden auto-submitting form (a classic CSRF pattern) using POST still won't carry the cookie.
None Sent on every cross-site request without restriction — requires the Secure flag to be set alongside it (modern browsers reject SameSite=None without Secure). Legitimate use case: a third-party embedded widget or payment iframe that genuinely needs your cookie sent in a cross-site context to function.
▸ the practical default

Lax is the modern browser default when no SameSite attribute is set at all, and it's the right choice for the overwhelming majority of session cookies — it blocks the cross-site POST pattern that powers most CSRF attacks while still letting ordinary link-clicking work the way users expect. Reach for Strict only for genuinely high-sensitivity actions, and None only when you have a real cross-site embedding requirement.

08

Multi-factor authentication

Confirmation across multiple distinct, independent factor categories.

Something you know

Passwords, PINs, security questions.

Something you have

Hardware keys, mobile authenticator apps (TOTP), SMS codes.

Something you are

Fingerprints, iris scans, facial biometrics.

▸ two passwords isn't MFA

Genuine multi-factor authentication requires factors from different categories above — a password plus a security question is still only "something you know," twice, and doesn't carry the same security guarantee as combining it with a genuinely independent factor like a hardware key or TOTP code.

09

Authentication attacks, mechanism by mechanism

Six named attack classes — here's exactly how each one actually executes.

Brute Force Attack

Programmatically attempting a vast permutation of random passwords against one account until a hit lands. Mitigated by lockout policies, rate limiting, and password complexity requirements.

Credential Stuffing

Feeding username/password pairs harvested from unrelated historical data breaches into a site's login form, betting on password reuse across services. Mitigated by MFA and breach-password-checking on signup.

Password Spraying

Testing one common password (e.g. Password123) against many different usernames — spreads attempts thin across accounts specifically to stay under per-account lockout thresholds.

Session Hijacking

Stealing a valid, already-authenticated session cookie via sniffing, XSS, or malware, then presenting it to impersonate that user — no password needed at all, exactly because of the Phase 05 equivalence.

Session Fixation

The attacker first obtains or sets a known session ID (e.g. via a URL parameter the app naively accepts before login, like ?PHPSESSID=attackerknown123), tricks the victim into authenticating while using that exact session ID, then — since the app never issued a fresh ID after login — uses the same known ID themselves to ride the now-authenticated session.

Weak Passwords

Default or trivially guessable credentials (123456, admin) that don't even require any of the above techniques — just a single guess.

▸ the single fix that defeats session fixation specifically

Session fixation only works because the application reused the pre-login session ID after authentication succeeded. The fix is structural: always regenerate the session ID immediately after a successful login, discarding whatever ID existed before. This single practice — distinct from hijacking defenses — closes the fixation attack entirely, regardless of how the attacker originally supplied the pre-login ID.

10

Best practices

The checklist version of everything above.

  • Enforce strong password complexity requirements
  • Enforce MFA everywhere it's feasible, not just for admin accounts
  • Set Secure, HttpOnly, and an appropriate SameSite value on every session cookie
  • Run full-site HTTPS coverage — no plaintext fallback for anything session-related
  • Implement strict, reasonable session timeouts (both idle and absolute)
  • Regenerate the session ID immediately after every successful login (Phase 09's fixation defense)
  • Implement account lockout or progressive throttling after repeated invalid login attempts
11

Hands-on lab — DVWA + Burp Suite

2–3 hours. Watch every concept above happen in real traffic.

  1. Open DVWA Start Apache and MySQL, navigate to http://127.0.0.1/DVWA, log in with admin / password.
  2. Intercept the login request Launch Burp Suite, toggle Intercept ON under the Proxy tab. Submit the login form and inspect the raw HTTP request — look at the POST body parameters carrying the username/password, and note any cookies already present before authentication even succeeds.
  3. Inspect the Set-Cookie response Examine the server's response to a successful login. Find the Set-Cookie header, record the exact PHPSESSID value, and note every attribute accompanying it.
  4. View cookies in the browser Open DevTools (F12) → Application/Storage → Cookies → your DVWA domain. Check the HttpOnly and Secure columns specifically.
  5. Explore session behavior Navigate across different DVWA pages and confirm the same cookie value persists across requests. Then log out and check: did the server actually invalidate that session ID, or does the old value still work if you replay it?
  6. Observe a failed login Submit deliberately wrong credentials. Note the HTTP status code and exact response text — would an automated brute-force script be able to parse this response to detect success vs. failure reliably?
  7. Identify security features (or their absence) Check whether this DVWA instance sets Secure, HttpOnly, or any SameSite value at all on its session cookie — DVWA is deliberately vulnerable by design, so don't be surprised to find these missing.
▸ a specific thing worth testing in step 5

Copy the PHPSESSID value before logging out, log out through the UI, then manually re-set that same cookie value in your browser and reload. If the session still works after "logout," that's a real, reportable finding — proper logout should invalidate the session server-side, not just clear the client-side cookie.

Troubleshooting

What actually goes wrong during this lab.

Burp Suite intercept never shows the login request

Confirm your browser's proxy settings actually point at Burp (typically 127.0.0.1:8080) and that Burp's own CA certificate is trusted if you're testing anything over HTTPS — for plain HTTP DVWA on localhost this usually isn't needed, but double-check Intercept is toggled ON, not just that Burp is running.

Can't find Set-Cookie in the response at all

If DVWA's session was already established from a previous login attempt, the server may not re-send Set-Cookie on a request that reuses an already-valid session. Clear cookies for the domain in your browser first, or use Burp's Repeater to manually resend the login request fresh.

Manually re-setting the old PHPSESSID after logout doesn't reproduce anything interesting

That's actually a valid, useful result — it means this particular DVWA configuration does properly invalidate sessions server-side on logout. Note that as a finding too; not every test is expected to find a vulnerability, and documenting "checked, not present" is itself a real skill.

Next steps

Where today's session model gets put under direct attack.

Today built the defender's mental model of how authentication and sessions are supposed to work. The natural next step is testing what happens when any one piece of it is missing or misconfigured — weak session ID entropy, a missing HttpOnly flag paired with an XSS injection point, or a CSRF attempt against a cookie left at SameSite=None without good reason. Each of those is a direct, practical extension of a specific section above, and exactly the kind of finding a real authentication security review is built to catch.
