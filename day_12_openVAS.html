00

Overview & architecture

What you're actually installing, and why it's five services pretending to be one tool.

"OpenVAS" is the name people use colloquially, but as of recent releases the project is officially the Greenbone Vulnerability Management (GVM) framework — OpenVAS Scanner is just the scanning engine component inside it. When you run apt install gvm, you're pulling in a small distributed system, not a single binary:

gvmd

The management daemon. Owns the PostgreSQL database, task scheduling, user accounts, report storage, and exposes the GMP (Greenbone Management Protocol) interface over a Unix socket. This is the brain.

openvas-scanner + ospd-openvas

The actual scanning engine, driven by NASL (Nessus Attack Scripting Language) plugins called NVTs (Network Vulnerability Tests). ospd-openvas is the wrapper daemon that speaks OSP (Open Scanner Protocol) so gvmd can hand it scan jobs.

gsad

Greenbone Security Assistant daemon — a small HTTPS server that renders the web UI you'll actually click around in, talking to gvmd over GMP behind the scenes.

Greenbone Data Feed (GVMD_DATA / NVT feed)

Not a service — a constantly-updated dataset of NVTs, CVEs, CPEs, CERT advisories, and scan configs, pulled via rsync from Greenbone's community feed servers. An unsynced install finds nothing. This is the step people skip and then wonder why their first scan returns an empty report.

▸ why this matters before you install

Every install headache you'll hit later traces back to one of these four pieces being mis-synced, mis-permissioned, or simply not finished initializing yet. Knowing the architecture turns "it's broken" into "gvmd's socket isn't up yet" or "the NVT feed is still syncing" — both fixable in seconds once you know where to look.

01

Lab prerequisites

Resource floor, disk, and network assumptions before you start.

  • RAM: 4 GB absolute minimum, 8 GB+ recommended. gvmd + PostgreSQL + the feed sync process are not light, and feed sync will swap-thrash a 2 GB VM into a coma.
  • Disk: 20 GB+ free. The NVT/CVE/CPE feed plus PostgreSQL data directory easily clears 6–8 GB once fully synced, and report storage grows from there.
  • CPU: 2 cores minimum. NASL script execution during a scan is CPU-bound, not just network-bound.
  • Network egress: outbound access to Greenbone's community feed servers (rsync over TCP) for the initial and ongoing syncs. If you're behind a restrictive lab firewall, this is the first thing to check when sync hangs.
  • A scan target that you own or are explicitly authorized to test. A second VM (e.g. Metasploitable2, a deliberately-vulnerable Debian box, or your own home-lab router on an isolated VLAN) — never a host outside your lab network.
▸ scope reminder

OpenVAS is an active scanner — it sends real exploit-adjacent probes, not just banner grabs. Running it against anything outside a network you own or have written authorization to test is the same legal exposure as running Metasploit against it. Keep this entirely inside your isolated lab segment.

02

Installation

Kali ships GVM in its repos as a metapackage — this is the path of least resistance. A from-source build is covered briefly at the end for non-Kali distros.

$ Update and install the metapackage

root@kali — bash
# refresh package lists and pull current upgrades first —
# gvm has a lot of dependency churn between Kali releases
┌──(root@kali)-[~]
└─# apt update && apt full-upgrade -y

┌──(root@kali)-[~]
└─# apt install gvm -y

This pulls in gvmd, gsad, openvas-scanner, ospd-openvas, postgresql, redis-server (used by ospd-openvas for scan result caching), and the gvm-tools / python3-gvm packages for scripting access later.

▸ disk space check

Before running setup, confirm free space with df -h /var. The feed lives under /var/lib/openvas/plugins and /var/lib/gvm — if your VM's disk was provisioned thin, this is where it bites you mid-sync.

$ Run the setup wizard

gvm-setup is the orchestration script that creates the PostgreSQL role and database, generates TLS certificates for gsad, builds the initial NVT/SCAP/CERT feed sync, creates the admin user, and starts every service in the right order. This is the long step — budget 30–90 minutes depending on your connection to the feed mirrors.

root@kali — bash
┌──(root@kali)-[~]
└─# gvm-setup

[*] Updating NVT feed...
# rsync pulls thousands of .nasl scripts — this is the slow part
[*] Updating CERT-Bund / DFN-CERT advisories...
[*] Updating SCAP (CVE/CPE) data...
[*] Generating certificates for gsad...

▸ write the generated password down now

gvm-setup auto-generates a random password for the admin user and prints it once, at the very end of a very long, scrollback-heavy run. If you miss it, don't panic — Phase 14 below covers resetting it with gvmd --user=admin --new-password=.... But save yourself the trip.

$ Verify the install with gvm-check-setup

This is the single most useful command in the entire toolchain. Run it after setup and after every future feed sync or reboot — it walks every dependency, socket, and permission gvmd needs and tells you exactly what's missing, instead of you guessing from a vague connection error later.

That last line is normal if you run the check mid-sync. Re-run it after the next phase confirms feed sync is complete — you're looking for an NVT count in the tens of thousands, not a handful.

Building from source on non-Kali / non-Debian distros

On Arch, Fedora, or a hardened distro without a maintained GVM package, you'll build from the Greenbone Community Edition source releases instead: gvm-libs → gvmd → pg-gvm → gsa (web assets) → gsad → openvas-smb → openvas-scanner → ospd-openvas, in that dependency order, each via the standard cmake -B build && cmake --build build && cmake --install build pattern. This is a multi-hour exercise in version-pinning and is genuinely easier inside a container — the official greenbone/gvmd and greenbone/openvas-scanner Docker images, composed with gvm-tools, sidestep this entirely if your lab allows containers. For this Day 11 lab, stick with Kali's metapackage; revisit source builds only once you need a production deployment outside Debian-family systems.

03

First boot & feed synchronization

Starting the services, and understanding the three independent feeds that all have to land before scans are meaningful.

$ Start (or restart) all services

root@kali — bash
┌──(root@kali)-[~]
└─# gvm-start

# under the hood this is equivalent to:
#   systemctl start postgresql redis-server@openvas
#   systemctl start ospd-openvas gvmd gsad

┌──(root@kali)-[~]
└─# systemctl status gvmd ospd-openvas gsad --no-pager

Three independent feed types sync on their own schedules and timelines — knowing which is which saves you from misdiagnosing "it's still syncing" as "it's broken":

FeedToolContainsTypical first-sync time
NVT feedgreenbone-nvt-syncThe actual NASL vulnerability-test scripts the scanner executes20–60 min
SCAP feedgreenbone-scapdata-syncCVE and CPE dictionaries used to cross-reference findings15–40 min
CERT feedgreenbone-certdata-syncCERT-Bund / DFN-CERT advisories linked to NVTs5–15 min

gvm-setup kicks all three off automatically, but on later days you'll want to refresh them manually (feed data changes daily upstream):

root@kali — bash
┌──(root@kali)-[~]
└─# runuser -u _gvm -- greenbone-nvt-sync
┌──(root@kali)-[~]
└─# runuser -u _gvm -- greenbone-scapdata-sync
┌──(root@kali)-[~]
└─# runuser -u _gvm -- greenbone-certdata-sync

# then tell gvmd to rebuild its internal NVT cache from the new data
┌──(root@kali)-[~]
└─# gvmd --rebuild

# finally bounce ospd-openvas so it picks up newly-synced NASL scripts
┌──(root@kali)-[~]
└─# systemctl restart ospd-openvas
▸ the most common "no vulnerabilities found" cause

Forgetting the gvmd --rebuild + ospd-openvas restart after a manual sync. The NASL files land on disk but neither daemon picks them up until told to. If a scan completes suspiciously fast with an empty report, this is step one to check — before you start suspecting your target or your network config.

04

Accessing the GSA web UI

gsad serves the Greenbone Security Assistant over HTTPS on a self-signed cert — your browser will complain, and that's expected in a lab.

  1. Confirm gsad is listening By default gsad binds to 127.0.0.1:9392. Check with ss -tlnp | grep 9392. If you need remote access from your host machine (common when Kali is a headless VM), restart gsad with gsad --listen=0.0.0.0 --port=9392 or set the listen address in /etc/default/gsad — but only do this on an isolated lab network, never on anything internet-facing.
  2. Navigate to the UI Open https://127.0.0.1:9392 (or your VM's IP) in a browser. Accept the self-signed certificate warning — this is gsad's own generated cert, not a sign of compromise, but in a real production deployment you'd replace it with a properly issued cert.
  3. Log in Username admin, password from the gvm-setup output (or your reset password from Phase 14). You'll land on the Dashboard — a set of empty charts until your first scan completes.
▸ orienting yourself in the UI

The top nav has five menus worth knowing immediately: Dashboard (summary charts), Scans (Tasks, Reports, Results, Vulnerabilities — where you'll spend 80% of your time), Assets (discovered hosts and operating systems, auto-populated from scan results), Resilience (remediation tickets and policy compliance), and Configuration (Targets, Port Lists, Credentials, Scan Configs, Alerts, Schedules — where you'll build everything before you can run a Task).

05

Core concepts before you click anything

GVM's object model is rigid and a little verbose by design — once it clicks, the UI stops feeling like a maze.

Target

A named definition of what to scan — host(s)/IP range, plus which port list to use and optional credentials. Reusable across multiple Tasks.

Port List

Which TCP/UDP ports to probe. Greenbone ships defaults ("All IANA assigned TCP", "All TCP and Nmap top 100 UDP") — you can also import nmap-style port range strings.

Scan Config

The how thoroughly — which NVT families are enabled, and per-family/per-NVT preference tuning (timeouts, intrusiveness, specific check toggles). "Full and fast" is the default workhorse.

Credentials

SSH/SMB/SNMP/ESXi login material for authenticated scanning — lets NVTs log in and check package versions and config files directly instead of guessing from the network.

Task

The executable unit: binds a Target + Scan Config (+ optional Schedule, Alert, Scanner) into something you can actually run. Running a Task produces a Report.

Report / Results / Vulnerabilities

A Report is one Task run's output. Results are individual findings within it. Vulnerabilities is a cross-report aggregated view — "every distinct vuln class ever found," useful for trend tracking.

The dependency order is strict: you cannot build a Task without a Target, and a Target needs at least a Port List before it'll save. Build bottom-up: Port List (use defaults to start) → Credentials (if doing authenticated scanning) → Target → Scan Config (use a default) → Task.

06

Defining targets

Configuration → Targets → the small star/plus icon (New Target).

  1. Name Something descriptive you'll recognize in six Tasks' time — e.g. lab-metasploitable2, not target1.
  2. Hosts Accepts a single IP, a comma-separated list, a CIDR block (192.168.56.0/24), or a hyphenated range (192.168.56.10-20). For this lab, point it at your isolated vulnerable VM's single IP.
  3. Exclude Hosts (optional) Carve out specific addresses from a range — handy when scanning a subnet but explicitly excluding your gateway or a fragile legacy device.
  4. Port List Default "All IANA assigned TCP" is a reasonable starting point. For a faster first pass, "Nmap top 100 TCP" trades coverage for speed — useful for an initial recon sweep before a full scan.
  5. Alive Test Controls how GVM decides a host is up before bothering to scan it (ICMP ping, TCP ACK, ARP, or combinations). On a host behind a firewall dropping ICMP, leaving this on "ICMP Ping" alone will make GVM falsely conclude the host is down and skip it entirely — set it to "Consider Alive" if you already know the target's status, or combine probe types.
  6. Credentials for authenticated checks (optional, covered in depth in Phase 09) SSH credential for Linux/Unix targets, SMB credential for Windows targets. Leave blank for an unauthenticated, externally-visible-only scan.
▸ the silent "0 results" trap

If Alive Test is misconfigured against a target that filters the relevant probe type, GVM marks the host dead and the Task completes near-instantly with an empty report — no error, no warning. A suspiciously fast "Done" status on a Task is almost always this, not an actually clean host.

07

Scan configs & profiles

How thorough — and how loud — the scan is. Pick based on what you're optimizing for, not just "the biggest one."

Discovery Host and service discovery only — port scanning and service/banner identification, minimal vulnerability checks. Use for a fast initial recon pass. ~ minutes
Host Discovery Just determines which hosts in a range are alive — no port or vuln scanning at all. Use before committing to a full scan across a large subnet. ~ seconds–min
System Discovery Discovery plus OS and hardware/software inventory detection — good for asset-mapping a new lab segment without firing intrusive checks yet. ~ minutes
Full and fast The default workhorse — runs the full NVT set but uses port/banner detection to skip checks that obviously don't apply, instead of brute-forcing every NVT against every port. Best default for this lab. 15 min – several hrs
Full and fast ultimate Same as above plus checks that can crash unstable services or devices (some DoS-adjacent NVTs are enabled). Never run against production or fragile embedded devices. longer, riskier
Full and very deep Maximum thoroughness — minimal skip-logic, every applicable NVT family probed regardless of detected services. Slowest, most exhaustive, most likely to generate noisy/intrusive traffic. hours+

For Day 11's lab exercise: start with Full and fast against your single vulnerable-VM target. It's thorough enough to be educational and fast enough to iterate on while you learn to read reports.

⚙ Customizing a scan config

Cloning a default config (rather than editing it directly) and adjusting NVT family toggles is the right pattern once you outgrow the presets — e.g. disabling the "Denial of Service" family entirely when scanning anything you can't afford to crash, or tightening timeout preferences for NVTs that hang against unusual targets.

08

Building & running a task

Scans → Tasks → New Task. This is where Target + Scan Config become an executable job.

  1. Name & target Pick a descriptive name and select the Target you built in Phase 06 from the dropdown.
  2. Scan Config Select Full and fast (or your custom clone) from Phase 07.
  3. Scanner Leave as OpenVAS Default unless you've registered an additional remote scanner — out of scope for this lab but relevant once you're managing scans across multiple network segments from one gvmd instance.
  4. Order for target hosts Sequential vs. random — matters mainly for large multi-host targets where you want predictable progress reporting (sequential) vs. spreading load less predictably (random).
  5. Maximum concurrently executed NVTs per host / Maximum concurrently scanned hosts The two knobs that actually control scan aggressiveness and duration. Lower both if you're worried about overwhelming a fragile target or a slow lab network link; raise both (cautiously) once you're scanning a multi-host range and want to parallelize.
  6. Save, then click the play (▶) icon next to the Task in the Tasks table to start it.
▸ watching it run

The Tasks table shows a live progress percentage while a scan runs. Click into the Task to see the in-progress Report build incrementally — results populate as NVTs complete, you don't have to wait for full completion to start reading early findings.

09

Authenticated / credentialed scans

The difference between "what an attacker sees from outside" and "what's actually patched" — and the single highest-value upgrade to your scan accuracy.

An unauthenticated scan can only infer software versions from banners, response fingerprints, and exposed service behavior — which means it both under-reports (can't see what's installed but not listening) and occasionally over-reports (a banner doesn't always mean what it says, especially on hardened or backported systems). A credentialed scan logs in and checks installed package versions, config files, and registry/file permissions directly.

+ Creating an SSH credential

Configuration → Credentials → New Credential. Type "Username + Password" or "Username + SSH Key" (key-based is strongly preferable — no plaintext password sitting in the GVM database). For a key-based credential, paste the private key and passphrase (if any); GVM only needs read access to run version-check commands, so a dedicated low-privilege scanning account on the target is good practice rather than using root/admin credentials.

+ Attaching it to a target

Edit your Target from Phase 06, and under SSH Credential select the credential you just created, plus the SSH port (default 22). For Windows targets, the equivalent is an SMB credential under the same Target edit screen.

▸ verifying it actually authenticated

Don't assume credentials worked just because the Task completed. Open the Report and check for a finding called "Authentication" or filter results by the SSH Authentication Configuration NVT — it explicitly reports whether the login succeeded. A failed silent auth still produces a report; it just degrades silently back to unauthenticated-equivalent coverage, which is easy to miss if you don't check.

10

Reading the report

CVSS severity bands, what each column actually means, and how to not drown in low-value findings.

SeverityCVSS v3 rangeTypical triage action
CRITICAL9.0 – 10.0Patch/remediate immediately — often remote code execution or full auth bypass.
HIGH7.0 – 8.9Prioritize this sprint — significant confidentiality/integrity/availability impact.
MEDIUM4.0 – 6.9Schedule remediation; assess exploitability in your specific context first.
LOW0.1 – 3.9Backlog — usually info disclosure or requires unusual preconditions.
LOG0.0Informational only — banner grabs, open-port notices, no actual finding.

▤ Report table columns

  • Vulnerability — the NVT's title/finding name.
  • Severity — the CVSS score and band, as above.
  • QoD (Quality of Detection) — a 0–100 confidence score for how reliable the detection method itself is. A version-check via authenticated package query scores near 100; a remote banner-string guess might score 70–80. Filter on QoD before triaging severity — a Critical finding with QoD 30 deserves more skepticism than a Medium with QoD 99.
  • Host / Location — which target and which port/service the finding applies to.
  • CVE(s) — cross-referenced identifiers from the SCAP feed, clickable through to full CVE detail including the NVD description and CVSS vector string.

▤ Filtering effectively

The filter bar accepts a compact query syntax — this is worth memorizing over clicking through dropdowns every time:

GSA report filter bar
# severity 7.0+ only, sorted by severity descending
severity>=7.0 sort-reverse=severity

# exclude pure informational/log-level noise
severity>7.0 sort-reverse=severity

# only high-confidence detections
qod>=70 severity>=4.0

# findings mentioning a specific CVE or product
"OpenSSH" severity>=4.0
▸ exporting reports

The Report view's download icon offers PDF, XML, CSV, and a few SCAP-aligned formats (ARF/CPE). XML is what you want if you're piping results into another tool or writing a parsing script — it's the full structured data, not the rendered summary.

11

Tuning & false-positive triage

Scanners are deliberately tuned toward recall over precision. Learning to triage is half the actual skill.

✓ The override workflow

Rather than deleting a finding you've determined is a false positive (which loses the audit trail), GVM supports Overrides: Scans → Overrides → New Override, scoped to a specific NVT, host, port, or Task, with a new severity (often "False Positive," which maps to a 0 severity with a note) and a free-text justification. On every future scan of the same target, the override automatically re-applies — your triage work compounds instead of repeating.

✓ Common false-positive patterns to recognize

Version-string false match

A backported security patch on an older-looking version string (common on Debian/Ubuntu LTS) trips a CVE check keyed purely on version number. Cross-check the vendor's actual changelog/CVE advisory before overriding — sometimes it's genuinely still vulnerable despite the backport.

Low QoD remote guesses

Anything detected via banner inference alone (QoD < 50) on a service known to lie about its version (common with hardened web servers) warrants manual verification, not blind trust.

Default-credential checks against changed creds

NVTs that try known default logins will correctly report "not vulnerable" once you've changed them — but a stale scan config run before a password rotation will show a now-incorrect finding. Re-scan, don't just override, when the underlying state has actually changed.

Service detected, vuln not actually reachable

A vulnerable library version flagged on a port that's firewalled from the actual scan source in production, but reachable in your flat lab network. Document the actual production exposure context in the override note — the technical finding is real, but risk depends on network topology.

▸ what not to do

Don't reflexively override every Critical/High finding that's "probably fine." Verify first — manually confirm with a targeted check (a quick nc, curl, or a manual exploit-module test against your own lab target) before downgrading severity. The override system's value depends entirely on the justifications behind it being accurate.

12

GVM via CLI & the GMP/XML API

Everything the web UI does is a GMP command underneath — useful once you want scripting, automation, or CI-pipeline integration.

$ gvm-cli — raw GMP over the Unix socket

gvm-cli (from the gvm-tools package) sends raw GMP XML commands directly to gvmd. Useful for quick one-offs and for understanding the protocol before scripting against it with the Python library.

root@kali — bash
# list all current tasks and their status
┌──(root@kali)-[~]
└─# gvm-cli --gmp-username admin --gmp-password 'your-password' \
    socket --xml "<get_tasks/>"

# get the full XML report for a specific report ID
┌──(root@kali)-[~]
└─# gvm-cli --gmp-username admin --gmp-password 'your-password' \
    socket --xml "<get_reports report_id='REPORT-UUID' format_id='a994b278-1f62-11e1-96ac-406186ea4fc5'/>"

$ python-gvm — scripted automation

For anything beyond one-off queries, python-gvm gives you a proper client object instead of hand-building XML strings. This is the realistic path toward "trigger a scan from a CI pipeline" or "auto-export new Critical findings into a ticketing system":

scan_lab_target.py
#!/usr/bin/env python3
"""Authenticate to gvmd over the Unix socket and kick off an existing Task."""
from gvm.connections import UnixSocketConnection
from gvm.protocols.gmp import Gmp
from gvm.transforms import EtreeCheckCommandTransform

connection = UnixSocketConnection(path="/run/gvmd/gvmd.sock")

with Gmp(connection, transform=EtreeCheckCommandTransform()) as gmp:
    gmp.authenticate("admin", "your-password")

    # find the task by name, then start it
    tasks = gmp.get_tasks(filter_string="name=lab-metasploitable2")
    task_id = tasks.xpath("task/@id")[0]

    response = gmp.start_task(task_id)
    report_id = response.xpath("report_id/text()")[0]
    print(f"Scan started, report id: {report_id}")
▸ where the format_id UUIDs come from

Report/output format IDs (like the PDF/XML format constant used above) are fixed UUIDs shipped with GVM — list them with <get_report_formats/> via gvm-cli rather than memorizing them; they're stable across versions but not worth hardcoding from memory.

13

Scheduling & automation

Turning one-off scans into a recurring posture-monitoring habit.

⏱ Schedules

Configuration → Schedules → New Schedule. Define a start time, recurrence (iCal-style RRULE under the hood — daily/weekly/monthly/custom), and optional duration cap. Attach it to a Task via the Task's edit screen. A weekly Sunday-night scan against your lab segment is a reasonable habit to build now, before you're managing more than one host.

✉ Alerts

Configuration → Alerts → New Alert. The genuinely useful pattern: trigger on Task completion, condition "Severity at least" set to High, method Email (requires a configured local MTA or relay) or HTTP/HTTPS GET (fire a webhook into something like a Slack incoming-webhook URL or your own listener). This turns GVM from "thing I remember to check" into "thing that tells me when something's actually wrong."

▸ a sane default alert

Condition: Severity at least → High. Trigger: Task run status changed to → Done. Method: HTTP GET to a webhook. This fires exactly when you want to know — new High/Critical found on scan completion — without paging you for every Log-level info finding.

14

Operational hardening

Treat the scanner itself as an asset worth protecting — it holds credentials to everything it scans.

  • Change the admin password immediately if you didn't capture the auto-generated one: gvmd --user=admin --new-password=<new-password> (run as the gvm service user, gvmd must be stopped or this targets the live DB directly depending on version — check current docs for your exact 22.x point release before running against a live service).
  • Create scoped, non-admin users for anyone else accessing the instance — Configuration → Users → New User, with role Observer (read-only) or User (can create/run own Tasks) rather than handing out admin broadly.
  • Restrict gsad's listen address to your lab's internal interface only, never 0.0.0.0 on anything with a routable/public IP.
  • Rotate scan credentials stored in GVM the same way you'd rotate any other privileged credential — they're encrypted at rest in PostgreSQL but are still a high-value target if the box is compromised.
  • Back up the PostgreSQL gvmd database before any major version upgrade — pg_dump against the gvmd database, stored somewhere outside the VM itself.
15

Troubleshooting

The failure modes you'll actually hit, in rough order of frequency.

gsad won't load / "connection refused" in browser

Check systemctl status gsad first. If it's not running, journalctl -u gsad -n 50 almost always shows a certificate path or permissions error from setup. Most common cause: gsad started before gvmd's socket existed — restart in order with gvm-stop && gvm-start rather than restarting gsad alone.

Scan completes instantly with zero results

Almost always one of: (1) Alive Test misconfigured against a target that filters the probe type — see Phase 06; (2) NVT feed still mid-sync or never finished — check gvm-check-setup's NVT count; (3) target genuinely unreachable from the scanner's network position — confirm with a manual ping or nmap from the same Kali host first.

"NVT collection... X NVTs found, which is too low"

The feed sync hasn't finished, or finished but gvmd --rebuild was never run afterward. Re-run the sync commands from Phase 03, then explicitly run gvmd --rebuild and restart ospd-openvas. Confirm progress with repeated gvm-check-setup runs rather than guessing from elapsed time.

Authenticated scan shows as failed/incomplete auth despite correct credentials

Check the target's SSH server config for PermitRootLogin / key-type restrictions if using a key-based credential, and confirm the scanning account actually has shell access (not just SFTP-chroot, which breaks the version-check commands NVTs run). Also confirm the Target's configured SSH port matches what's actually listening — a non-default SSH port left at "22" in the Target config is a common silent miss.

PostgreSQL connection errors after a reboot

PostgreSQL sometimes starts slower than gvmd expects on boot. Check systemctl status postgresql is fully active before gvm-start; if you've enabled gvmd as a boot-time service, consider adding an explicit systemd dependency/ordering (After=postgresql.service) via a drop-in unit rather than fighting race conditions manually each reboot.

Feed sync hangs or times out

Confirm outbound rsync (TCP) isn't blocked by your lab's firewall/NAT setup — this is the most common cause inside restrictive corporate or campus networks. Test with a direct rsync --list-only rsync://feed.community.greenbone.net:/nvt-feed style probe (check current feed hostnames against official docs, as Greenbone has migrated feed endpoints before) to isolate network vs. application-layer failure.

Where this fits / next steps

Connecting Day 11 forward to the rest of the roadmap.

OpenVAS/GVM gives you the known-vulnerability half of a security assessment — matching what's installed against what's been publicly disclosed. It deliberately does not cover:

  • Exploitation — confirming a flagged CVE is actually exploitable in your specific configuration is Metasploit/manual-PoC territory, not GVM's job.
  • Logic flaws & business-logic vulnerabilities — anything that isn't a known CVE against a known package version needs manual testing or a dedicated tool (Burp Suite for web app logic, for instance).
  • Behavioral/anomaly detection — this is where the ML-based IDS work from later in the roadmap picks up; GVM tells you what's vulnerable, an IDS tells you what's actively being attacked.

A natural follow-on lab exercise: export this scan's XML report, and write a small Python parser (using lxml or xml.etree) that extracts every Critical/High finding's CVE ID and feeds it into a simple deduplicated tracking sheet — the same pattern you'd use to wire GVM output into a real ticketing pipeline later.
